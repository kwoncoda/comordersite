## 1. 서비스 개요

### 1.1 서비스명·정체성

| 항목 | 값 |
|---|---|
| 서비스명 | **오늘 저녁은 치킨이닭!** |
| 부제목 / 보조 카피 | **WINNER WINNER CHICKEN DINNER!** (PUBG 승리 메시지 차용) |
| 별칭 | 치킨이닭 / 오저치 |
| 운영 주체 | **컴퓨터모바일융합과** (학생회) |
| 마스코트 | 군용 헬멧·아머·소총을 든 흰 곰 캐릭터 (`웹 로고.png`) |
| 배지 톤 | "**#1/100**" 배틀로얄 식별자 |

### 1.2 한 줄 소개

> "축제 부스에서 주문 외치지 말고, 폰으로 주문하고 번호로 받아가세요."

### 1.3 서비스 목적과 운영 일정

**목적**
- 종이/구두 주문에서 발생하는 **누락·오기·말다툼** 제거
- 운영진이 **주문 현황을 한 화면으로** 파악
- 축제 종료 후 **정산을 1시간 내** 끝낼 수 있도록 자료 자동 집계
- 결제 시스템 도입 부담 없이 **계좌이체 + 사용자 신고 + 운영진 확인**으로 운영
- 사용자가 **자기 주문이 어디까지 진행됐는지** 직접 확인 가능 (조리 현황판)
- 축제 종료 후 **데이터 손실에도 정산 근거 유지** (ZIP 백업)

**운영 일정 (G7)**
- **2026-05-20(수)** 16:30 오픈 ~ 마감 (당일 상황 따라)
- **2026-05-21(목)** 16:30 오픈 ~ 마감 (당일 상황 따라)
- 마감 시각 가변 → **관리자가 "오늘 정산 마감" 버튼**을 직접 눌러 트리거 (`§12.1`)

### 1.4 해결하려는 문제

| 현장 문제 | 본 시스템의 해결 방식 |
|---|---|
| 큰 소리 주문이 시끄럽고 헷갈림 | 폰에서 메뉴 선택 → 주문번호 발급 |
| 종이 주문서 분실·이중 처리 | 모든 주문을 DB 1곳에 모음 |
| 메뉴 품절을 늦게 알림 | 운영진 토글 → 사용자 화면 즉시 반영 |
| 정산할 때 종이 영수증 더미 | 자동 집계 화면 + 관리자 수동 트리거 |
| 1학년 환영 쿠폰 운영 어려움 | 학번 prefix `202637` 검증 + 같은 학번 1회 사용 제한 (ADR-019) |
| 입금 매칭이 어려움 | 사용자가 직접 이름·은행 신고 → 운영진 통장 대조 |
| **사용자가 "조리 완료됐나?" 모름** | **조리 현황판 — 사용자가 주문번호로 자기 진행 상황 직접 확인** |
| **DB 장애·서버 종료 시 정산 근거 소실 위험** | **정산 마감 시 전체 데이터 ZIP 백업 다운로드** |

### 1.5 축제 현장에서 기대되는 효과

- 주문 한 건 처리 시간 **30초 → 10초** 단축 목표
- 주문 누락률 **5% → 1% 미만**
- "제 주문 언제 나와요?" 문의 **수십 건 → 0건 가까이** (조리 현황판이 흡수)
- 정산 소요 시간 **반나절 → 1시간**
- 운영진 목 갈라지는 일 0건

### 1.6 디자인 컨셉

| 축 | 참고 | 적용 방향 |
|---|---|---|
| **UX (사용성)** | **배달의민족** | 메뉴 카드 그리드, 상단 가로 분류 탭, 큰 카트 버튼, 단계 최소화, 모바일 폭 360~430px 최적화 |

**카피 톤 매칭:**
- 주문 완료 시 "**WINNER WINNER CHICKEN DINNER!**" 축하 문구

## 3. 핵심 기능 정리

### 3.1 사용자용 기능
- 메뉴 목록 보기 (분류·추천·**실시간 인기 랭킹**·품절 표시)

## 4. 사용자 주문 흐름

### 4.1 단계별 흐름

```
1. 사이트 접속 (QR 또는 단축 URL) → "오늘 저녁은 치킨이닭!" 랜딩
2. 약도/위치 미니 배너 (선택)
3. 메뉴 화면 진입
4. 분류별 메뉴 탐색 (치킨/사이드/음료)
   ├─ 사장님 추천·실시간 인기 메뉴 상단 노출
   └─ 품절 메뉴 비활성 + 표시
5. 메뉴 선택 → 수량 → 장바구니
6. "주문하기"
7. 수령 방법 선택 (매장 / 포장)
   └─ 매장 선택 시 테이블 번호 입력
8. **학번+이름 입력 (필수, ADR-021)**:
   - 기본: 학번 9자리 + 이름
   - 또는: "학번 없음" 체크박스 → 이름만 (외부인)
   - 쿠폰 사용 시: 학번 입력 자동으로 prefix `202637` 검증·used_coupons 중복 검증 (ADR-019)
9. 최종 주문 확인 — **금액은 서버가 메뉴·쿠폰·수량으로 자체 계산하여 표시** (ADR-020 Pattern B)
10. "주문 접수" → 주문번호 발급 → 상태: 주문 중
11. 주문 완료 화면
    ├─ 주문번호 (도그태그 스타일, 큼지막)
    ├─ "WINNER WINNER CHICKEN DINNER!" 축하 카피
    ├─ 계좌번호 + 최종 금액
    ├─ 주의사항
    ├─ "조리 현황 보기" 버튼 (조리 현황판 진입)
    └─ "이체했어요, 확인 요청 보내기" 버튼
12. 사용자가 본인 은행앱에서 이체 (사이트 외부)
13. 사이트 복귀 → "확인 요청 보내기"
14. 이체 확인 요청 화면 → **은행 입력** (이름은 주문 시점에 이미 있음, ADR-021) + 필요 시 "다른 이름으로 이체" 체크박스 → 상태: 이체 확인 요청
15. 운영진이 통장 대조 → 상태: 이체 완료
16. 조리 시작 → 사용자가 조리 현황판에서 "치킨이 기름 속으로 입장" 단계 확인
17. 조리 완료 → 현황판에 "수령 가능해요!" 표시 + 부스에서 주문번호 호명
18. 수령 → 완료
```

### 4.2 헷갈릴 수 있는 부분 + 개선

| 헷갈림 포인트 | 개선 |
|---|---|
| 계좌이체를 깜빡함 | 주문 완료 화면에서 **계좌번호 복사 버튼** |
| 이체 후 "확인 요청"을 깜빡함 | **빨간 강조 버튼** + "확인 요청 안 누르면 조리 시작 안 됨" 안내 |
| 입력한 이름과 실제 이체 이름이 다름 | "**은행앱에 표시되는 본인 이름 그대로** 적어주세요" 안내 |
| 은행 선택 누락 | 주요 은행 콤보박스 + "기타" 직접 입력 |
| 테이블 번호 미입력 | 매장 선택 시 미입력이면 다음 단계 비활성 |
| 쿠폰 대상자 아닌데 시도 | "이번 쿠폰은 컴퓨터모바일융합과 1학년만 사용 가능합니다" |
| **"조리 완료됐나?" 궁금함** | **조리 현황판** — 주문번호 입력해 현재 단계 직접 확인 (호명 안 듣고도 안심) |

> 약도 화면은 **사용자가 추후 제공할 약도 이미지**로 채움.

## 5. 관리자 운영 흐름

### 5.1 단계별 흐름 (본부 담당자 기준)

```
1. 관리자/본부 페이지 접속 (PIN)
2. 신규 주문 알림 — 상태: 주문 중
3. 사용자 "확인 요청" → 카드가 "이체 확인 요청"으로 이동
4. 본부 담당자 카드 클릭 → 상세
   - 사용자 입력 이름·은행·금액·시각 표시
   - 통장/카뱅앱과 대조
5. 매칭 시 "이체 확인" → 상태: 이체 완료
   매칭 실패 시 "보류" + 사유 메모
6. 조리 담당자 "조리 시작" → 상태: 조리 중 (사용자 현황판 갱신)
7. 조리 완료 → 상태: 수령 대기 (현황판에 "수령 가능" 표시)
8. 수령 담당자 "전달 완료" → 상태: 완료
9. 취소 요청 시 → 사유 선택 후 "취소"
10. 운영 중 주기적으로 판매 현황 확인
11. 마감 시각 도달 → 관리자가 **"오늘 정산 마감" 버튼** 클릭
12. 정산 화면 → **"전체 데이터 ZIP 백업 다운로드"** 클릭 → 학생회 대장 보관
```

### 5.2 실수 방지를 위한 화면·정보

| 위험 | 방어 |
|---|---|
| 같은 주문을 두 번 만듦 | 주문번호 고유, "조리 시작" 1회 활성 후 비활성 |
| 이체 확인 안 된 주문에 음식 만듦 | "이체 확인 요청" 카드 빨간 테두리, "이체 완료" 전 조리 시작 비활성 |
| 품절 메뉴를 계속 받음 | 사용자 화면 즉시 반영 + 관리자 메뉴 관리 표시 |
| 주문 상태 토글 깜빡 | 같은 상태 5분 이상 → 노란 경고, 10분 이상 → 빨간 + 알림 음 |
| 운영진 교대 시 인수인계 누락 | 본부 화면 우측 상태별 카운터(주문중/확인요청/이체완료/조리중/수령대기) |
| 이체 확인 요청 처리 지연 | 3분 이상 → 강조 알림 |
| 정산 버튼 누르기 전 시스템 종료 | 마감 체크리스트(`§15.1`)에 "정산 마감 + ZIP 백업" 필수 항목 |
| **정산 후 DB 장애로 데이터 소실** | **ZIP 백업 다운로드 후 학생회 USB·드라이브에 즉시 보관** |

## 6. 주문 상태 기획

### 6.1 상태 정의 (G5 결정 반영)

| # | 상태 | 한글 표기 | 진입 조건 | 다음 가능 상태 |
|---|---|---|---|---|
| 1 | `ORDERED` | **주문 중** | 사용자가 주문 확정 (이체 전) | 이체 확인 요청, 취소 |
| 2 | `TRANSFER_REPORTED` | **이체 확인 요청** | 사용자 "확인 요청" | 이체 완료, 보류, 취소 |
| 3 | `PAID` | **이체 완료** | 운영진 통장 대조 후 확인 | 조리 중, 취소 |

| 7 | `CANCELED` | **취소** | 운영진/사용자 취소 | (종료) |

## 7. 쿠폰 기능 기획 (ADR-019)

### 7.1 쿠폰 정책

| 항목 | 값 |
|---|---|
| 대상 | **컴퓨터모바일융합과 1학년** (학번 `202637XXX`) |
| 정식 명칭(카피) | **컴공 새내기 환영 쿠폰** (또는 **컴모융 1학년 인증 미션**) |
| 할인액 | 1,000원 |
| 사용 횟수 | 학번당 1회 (축제 전체 기간 공유, ADR-018) |
| 유효 기간 | 축제 기간 (5/20 ~ 5/21) |
| 검증 | **학번 prefix `202637` + 같은 학번 1회** (ADR-019, 명부 미도입) |
| 명부 | **미사용** — used_coupons 테이블이 사후 기록 역할 |

### 7.2 사용자 입력
- 학번 (9자리 숫자, 형식: `202637XXX`)
- 이름 (한글, 식별·기록 용 — 검증 X, 비어있지 않으면 통과)

### 7.3 검증 로직

```
입력 (학번, 이름) →
  Step 1: 학번이 9자리 숫자인가? (정규식 ^\d{9}$)
            → No: 거부 (형식 오류)
  Step 2: 학번이 "202637"로 시작하는가?
            → No: 거부 (1학년 아님)
  Step 3: 이름이 비어 있지 않은가? (한글 1자 이상)
            → No: 거부 (이름 누락)
  Step 4: 이 학번이 이미 used_coupons 테이블에 있는가?
            → Yes: 거부 (중복 사용)
  All Pass → used_coupons 레코드 생성, 1,000원 할인 적용
```

### 7.4 중복 방지
- `used_coupons` 테이블의 `학번` 컬럼이 **unique key**
- 같은 학번 재시도 거부 (이름·시점·주문 무관)
- 5/20과 5/21 양일 공유 (ADR-018)

### 7.5 안내 문구 (PUBG 톤)

| 상황 | 문구 |
|---|---|
| 학번 형식 오류 | "학번은 9자리 숫자입니다. 다시 입력해 주세요." |
| Prefix 불일치 (1학년 아님) | "이번 쿠폰은 **컴퓨터모바일융합과 1학년 환영 쿠폰**입니다. 1학년 학번이 맞는지 다시 확인해 주세요." |
| 이름 누락 | "이름을 입력해 주세요." |
| 이미 사용 | "이 학번은 이미 쿠폰을 사용했어요. 즐거운 축제 보내세요!" |
| 사용 성공 | "**컴공 새내기 인증 완료!** 1,000원 할인 적용 🎉" |

### 7.6 관리자 확인 정보 (명부 관리 화면 미도입)
- **쿠폰 사용 내역** 화면: 학번·이름·사용시각·주문번호·할인액
- **거부 시도 로그**: 사유별 카운트 (형식 오류 / prefix 불일치 / 이름 누락 / 중복 사용)
- **요약 카드**: "오늘 쿠폰 사용 인원: N명 / 총 할인액: N원"

### 7.7 개인정보 최소화
- 수집: 학번·이름·사용시각만 (사용 시점에 즉시 used_coupons에 기록)
- 보관: 축제 종료 + N일(권장 7~14일) 내 자동 삭제
- 동의: 쿠폰 입력 화면 체크박스 필수
- 사전 명부가 없으므로 "사용하지 않은 사용자의 학번/이름은 시스템에 존재하지 않음" — 개인정보 추가 보호

### 7.8 위조 위험 (Trade-off, ADR-019)
- ⚠️ 다른 학년이 `202637`로 시작하는 임의 9자리 숫자 + 임의 이름 입력 시 통과 가능
- ✅ 같은 학번 1회 제한으로 같은 위조 학번 반복 사용 X
- ✅ 1,000원 할인 수준이라 위조 incentive 제한적
- 운영진은 사후 used_coupons 내역을 학생증 명부와 대조 가능 (명목적 억제력)
- 더 강한 검증(예: 학번 마지막 3자리 범위 검증)이 필요하면 Phase 2에서 추가

## 8. 대기 시간 기능 (미도입)

이번 버전은 **대기 시간을 표시하지 않는다.** 자동 계산·운영진 토글·단순 안내·혼잡도 단계 표시 모두 도입하지 않는다.

**근거 (`ADR-010`, D8 재확인 2026-05-02):**
- 운영진 토글 누락 위험 제거
- 시스템 분기·UI 단순화
- 마감 시각 가변(`G7`)이라 시간 약속이 어려움
- 사용자 기대 관리는 "**조리 현황판**(`§10.4`)이 흡수" — 시간 약속 안 하고 단계만 알려줌

**대안 운영 권장:**
- 부스 입구에 칠판/포스터로 "주문 폭주 시 조리 현황판을 확인해 주세요" 정적 안내
- 매우 길어질 경우 운영진이 부스에서 직접 사과 안내

## 9. 메뉴 화면 기획

### 9.1 메뉴 분류 방식
- 상단 가로 탭: `전체` `🔥 인기` `⭐ 추천` `치킨` `사이드` `음료`
- 모바일 가로 스크롤

### 9.2 사장님 추천 표시
- 카드 좌상단 **⭐ 사장님 추천** 배지 (PUBG 톤이면 군용 스탬프 "RECOMMENDED")
- "추천" 탭에서 모아 보기
- **3개 이내** 권장

### 9.3 인기 메뉴 표시 (실시간 랭킹 + 동적 카피)

**랭킹 섹션:**
```
🔥 실시간 인기 메뉴 TOP 3
1위 뿌링클        37개 판매
2위 양념치킨      29개 판매
3위 감자튀김      21개 판매
```

**동적 카피 (랭킹 변동에 따라 자동 생성):**

| 상황 | 카피 예시 |
|---|---|
| 1위가 2위 격차 큼 | "현재 **뿌링클**이 압도적 1위!" |
| 1위·2위 격차 작음 | "**양념치킨**이 추격 중입니다." |
| 첫 운영 (데이터 부족) | "🔥 학생회 추천 BEST" |
| 신메뉴 / 막 떠오르는 메뉴 | "**감자튀김** 빠른 상승세 🚀" |

**구현 정책:**
- 5분/10분 단위 갱신 (사용자 화면 자동 새로고침 또는 풀투리프레시)
- 카피는 규칙 기반 자동 생성 (1위 격차, 신규 판매 속도 등으로 분기)
- 운영진이 카피 룰을 ON/OFF 가능 (편향 우려 시)

> **목표:** 사회적 증거(social proof)로 메뉴 선택 가속화. 축제 분위기에선 "남들이 많이 먹는 거"가 강한 영향.

### 9.4 품절 메뉴 표시
- 카드 흐림 처리 + "**SOLD OUT**" 빨간 도장 스탬프
- 클릭 비활성

### 9.5 메뉴 가격 표시
- `4,000원` 형식 (천 단위 콤마)
- 가격 폰트는 메뉴명만큼 명확히

### 9.6 메뉴 이미지 정책
- 기본: 사용자 제공 사진
- Fallback: **이모지 + 컬러 배경** (또는 **마스코트 곰** 활용)
  - 치킨 → 🍗 / 노란
  - 사이드 → 🍟 / 주황
  - 음료 → 🥤 / 파랑
- 이미지 비율 통일 (4:3 또는 1:1), 권장 ≤ 200KB

### 9.7 세트 메뉴
- 미도입 (자유 조합으로 충분)

### 10.4 조리 현황판 (신규, MVP)

> 사용자가 주문번호로 자기 주문의 현재 단계를 폰에서 직접 확인. 시간 약속 X, 단계 + 귀여운 카피 O.

**진입 방법:**
- 주문 완료 화면의 "🍗 조리 현황 보기" 버튼
- 주문번호 + (간이 인증 토큰)으로 URL 북마크 가능 → 폰 닫고 다시 열어도 접근

**상태별 카피 매핑:**

| 시스템 상태 | 사용자 화면 단계 | 카피 |
|---|---|---|
| `ORDERED` | 주문 접수 완료 | "주문 받았어요! 입금만 확인되면 바로 출동합니다." |
| `TRANSFER_REPORTED` | 입금 확인 중 | "입금 확인 중이에요. 잠시만 기다려 주세요 👀" |
| `HOLD` | 입금 정보 확인 필요 | "입력하신 이체 정보와 통장이 안 맞아요. 정보 수정 후 다시 보내주세요." |
| `PAID` | 조리 준비 중 | "🍗 입금 확인 완료! 곧 치킨이 출동합니다." |
| `COOKING` | 조리 중 | "🔥 지금 치킨이 기름 속으로 입장했습니다! 맛있게 튀겨지는 중이에요." |
| `READY` (음료/사이드만 남음) | 마무리 중 | "🥤 음료까지 챙기는 중이에요." (음료/사이드 포함 주문일 때) |
| `READY` | 수령 가능 | "✅ **#17번, 수령 가능해요!** 부스에서 호명을 들어주세요." |
| `DONE` | 완료 | "🎉 **WINNER WINNER CHICKEN DINNER!** 맛있게 드세요." |
| `CANCELED` | 취소됨 | "주문이 취소되었습니다. 부스에 문의해 주세요." |

**시각 표현:**
- 진행 단계 progress bar (5단계: 접수 → 입금확인 → 조리 → 마무리 → 수령)
- 마스코트 곰 캐릭터가 단계별로 변형 (헬멧 쓰는 중 / 출동 중 / 치킨 가져오는 중 / 도착 등)
- **SSE(Server-Sent Events)로 단계 변경 즉시 push** (1A 결정, 2026-05-03) — 사용자가 새로고침 안 해도 서버가 상태 바뀌면 그 순간 화면 갱신. 연결 끊어지면 브라우저가 자동 재연결.

**인증·보안 (D17 closed, ADR-021 반영):**
- **학생 (학번 보유):** `학번(9자리) + 주문번호`로 본인 주문만 조회. 학번을 모르면 타인 주문 조회 불가 → 도그태그 SNS 공유에도 안전.
- **외부인 ("학번 없음" 체크):** 학번 없으므로 동명이인 충돌 위험. 주문 시 발급한 **토큰(쿠키 + URL)** + 주문번호로 조회. 토큰 만료 24시간 권장.
- 폰번호 뒷자리 보조 키 미도입 (PII 추가 부담만 있고 보안 추가 효과 미미)
- D17은 ADR-021/ADR-015 보강으로 closed

## 12. 정산 기획

### 12.1 정산 트리거 정책 (G7 + G8 + ADR-012)

- 마감 가변 → 자동 마감 X
- 관리자가 **"오늘 정산 마감" 버튼**을 눌러 일자별 정산 스냅샷 생성
- **진행 중 주문 가드 (G8, 2026-05-03)**: `ORDERED`/`TRANSFER_REPORTED`/`PAID`/`COOKING`/`READY`/`HOLD` 상태가 1건이라도 있으면 **마감 거부** + 팝업 안내:
  > "진행 중 주문 N건 (주문중 X / 이체확인요청 Y / 이체완료 Z / 조리중 W / 수령대기 V / 보류 U)이 있습니다. 모든 주문이 완료(또는 취소)된 후 다시 시도해 주세요."
  - 모든 주문이 `DONE` 또는 `CANCELED` 상태일 때만 마감 가능
  - **강제 마감·"마감 후 보류" 상태 미도입** (개념 자체 제거)
- 일자별 정산 + 합산 정산 화면 제공
- **정산 마감 직후 ZIP 백업 다운로드 권장** (`§12.5`)

### 12.2 정산에 모아야 할 정보

| 항목 | 계산 |
|---|---|
| 일자 | 정산 마감 누른 날짜 |
| 총 주문 수 | `DONE` 개수 |
| 총 매출 | `DONE` 합계(할인 후) |
| 쿠폰 할인 총액 | 할인액 합 |

| 메뉴별 판매 수량 | 메뉴별 `DONE` 카운트 |
| 메뉴별 매출 | 메뉴별 (수량 × 단가) |
| 취소 주문 수 | `CANCELED` 개수 |
| 보류 미해결 | `HOLD` 잔여 (마감 전에 모두 해결되어야 함, G8) |
| 포장/매장 비율 | 비율 |

### 12.5 ZIP 백업 다운로드 (신규, MVP)

> DB 장애·서버 종료에도 정산 근거를 학생회 대장이 보관할 수 있는 안전망.

**ZIP 파일 구조:**
```
chickenedak-backup-2026-05-20-2142.zip
├── manifest.json            (생성 시각, 일자, 주문 수, 매출 합계, 해시)
├── orders.json              (모든 주문 raw — 메뉴·수량·상태·이체정보)
├── orders.csv               (엑셀로 보기 좋은 평탄화 버전)
├── coupons.json             (사용된 쿠폰 학번·이름·시각)
├── coupons.csv
├── menu-snapshot.json       (정산 시점의 메뉴·가격·품절 여부)
├── settlement-summary.pdf   (§12.3 화면을 PDF로 캡처)
├── images/                  (메뉴 이미지 사본 — 추후 보고서 작성용)
│   ├── chicken-fried.png
│   └── ...
└── README.txt               (파일 구조 설명, 보관 권장사항)
```

**생성 흐름:**
1. 관리자가 "📦 ZIP 백업 다운로드" 버튼 클릭
2. 서버가 그 시점의 일자 데이터를 모아 ZIP 패키징 (수 초~수십 초)
3. 다운로드 시작
4. 서버에 다운로드 이력 기록 (정산 화면에 표시 — 누가/언제 받았는지)

**보안·개인정보:**
- ZIP에는 **쿠폰 학번·이름**, **이체 신고 이름·은행** 등 개인정보 포함
- 다운로드 받는 즉시 학생회 USB·암호화 폴더에 보관, 다운로드 자체는 관리자 권한 PIN 필요
- ZIP 파일 자체에 **간단한 패스워드 보호** 옵션 제공 (학생회장이 관리하는 패스워드)
- 정산 종료 + N일(권장 7~14일) 후 운영진이 사본 모두 폐기

**의존성:**
- 서버 측 `archiver` 또는 `JSZip` 같은 라이브러리 (ZIP 생성)
- PDF는 정산 화면 HTML을 서버 측 렌더링 → PDF 변환 (puppeteer 등) 또는 클라이언트 측 `html2pdf` (구현 시 결정)

### 12.6 자동 ZIP 스냅샷 (ADR-022)

수동 정산 ZIP의 안전망. 운영 중 30분 주기로 자동 ZIP을 Docker volume의 `backups/` 디렉터리에 떨굼.

**스냅샷 구조:** §12.5 ZIP과 동일 (단, `settlement-summary.pdf`는 정산 마감 전이라 생략).

**회전 정책:**
- 누적 6개만 유지 (3시간치)
- 가장 오래된 것 자동 삭제

**디스크 사용량:** 평균 100KB-1MB × 6 = 6MB max (무시 수준).

**사용 시나리오:**
- 정산 마감 전 서버·디스크 사고 발생 시 가장 가까운 자동 스냅샷으로 복구 (최악 30분 손실)
- 정산 화면에 자동 스냅샷 이력 표시 (수동/자동 구분)
- 운영 가이드: 사고 시 `backups/` 폴더의 최신 자동 스냅샷을 USB·암호화 폴더에 즉시 복사

**보안:** §12.5와 동일 정책 적용 (PII 포함, Docker volume 권한 관리, 정산 후 N일 폐기).

## 13. 운영 시나리오

### 13.1 주문이 적을 때
- 알림 음만 들리게
- 추천/인기 메뉴 노출
- SNS 공유 마케팅

### 13.2 주문이 몰릴 때 (별도 모드 X)

> **별도 폭주 모드 미도입 (`ADR-007`).** 일반 운영 흐름 연장선에서 처리.

권장:
- 본부 담당자가 "이체 확인 요청" 카드 우선
- 조리 부족 시 수령 담당자 일시 합류
- 사용자에겐 추가 안내 없음 — **조리 현황판이 사용자 안심감을 흡수**

### 13.3 메뉴 품절
- 조리 담당자 → 본부 알림 → 즉시 품절 토글
- 사용자 화면 즉시 비활성 + "**SOLD OUT**" 스탬프
- 이미 주문된 건은 부스에서 "대체 메뉴 / 환불" 안내

### 13.4 이체 신고와 통장 안 맞을 때
- 본부 담당자: "보류" + 사유 메모
- 사용자: 카드 상태 "보류" + **조리 현황판에서 "입금 정보 확인 필요" 카피 노출**
- 사용자가 정보 수정 후 재제출 → `TRANSFER_REPORTED` 복귀
- 끝까지 안 맞으면 부스 직접 방문

### 13.5 쿠폰 prefix 불일치 또는 중복 사용 시도
- 자동 거부 + 사유별 로그 (형식 오류 / prefix 불일치 / 이름 누락 / 중복 사용)
- 같은 IP 5회 초과 시도 → 일시 잠금
- 거부된 학번/이름은 used_coupons에 기록되지 않음 (실제 사용 시점에만 기록)

### 13.6 주문 취소
- 사용자 요청 → 부스 확인 → 시스템 취소 토글
- 이체 완료 건은 운영진 수동 환불
- 사유: "사용자 요청" / "재료 소진" / "오작 주문" / "기타"

### 13.7 운영진이 상태 변경 깜빡함
- 같은 상태 5분 → 노란 경고
- 10분 → 빨간 + 알림 음
- 이체 확인 요청 3분 → 강조

### 13.8 마감 시각이 다가올 때
- 관리자가 현장 분위기 보고 결정
- **새 주문 수신 차단 토글** (이 시점부터 신규 주문 안 받음)
- 진행 중인 주문(주문중/이체확인요청/이체완료/조리중/수령대기/보류)이 **모두 완료(또는 취소)될 때까지** 운영 유지
- 진행 중 0건 확인 후 **"오늘 정산 마감" 버튼** 클릭 → 강제 마감은 없음, 진행 중 있으면 가드 팝업으로 마감 거부
- 정산 화면 진입 → **"ZIP 백업 다운로드"** 클릭

### 13.9 사용자가 조리 현황판을 사용할 때 (신규)
- 조리 현황판은 **SSE 기반 push** (1A 결정, 2026-05-03)
- 서버는 상태 변경(예: `COOKING` → `READY`) 시 해당 주문의 SSE 채널로 이벤트 전송
- 클라이언트는 `EventSource`로 자동 재연결 — 와이파이 끊어졌다 돌아와도 다시 붙음
- 동시 SSE 연결 30개 이하라 서버 메모리·소켓 부담 무시 수준 — 모니터링만 두고 진행
- 부담 커지면 Phase 2에서 WebSocket으로 확장 (대시보드 실시간 동기화 등 추가 시)

### 13.10 ZIP 백업 다운로드 중 서버 장애 (신규)
- 다운로드 재시도 가능 (서버에 일시 캐시)
- ZIP 생성 트랜잭션 실패 시: 운영진에게 즉시 알림, 수동 재시도 버튼

### 13.11 서버 프로세스 장애 (Docker 자동 복구, ADR-023)
- **Docker `restart: always` 정책으로 자동 복구.** 컨테이너가 죽어도 5초 안에 자동 부활.
- 본부 담당자는 화면이 멈추거나 새 주문이 안 들어올 때 즉시 시스템 응답 확인
- 자동 복구 실패 시 수동 재시작 절차:
  1. 시스템 무응답 확인 (브라우저 새로고침 시 응답 없음)
  2. 호스트 머신 콘솔에서 `docker compose ps` 로 컨테이너 상태 확인
  3. `docker compose restart` (5초)
  4. 로그 확인은 `docker compose logs -f` (Tail)
  5. SSE 연결은 사용자 측 `EventSource`가 자동 재연결
  6. 진행 중 주문 상태는 Docker volume의 DB에 안전 보관 — 데이터 손실 없음
- 추가 안전망: ADR-022 자동 ZIP 스냅샷이 30분마다 떨어져 있어 최악값도 30분 데이터 손실로 제한
- 운영 가이드에 "호스트 콘솔 화면에 `docker compose logs -f` 항상 표시"·"본부 담당자 1명은 항상 화면 확인" 명시
- 정산 마감 가드(T3)는 살아있음 — 장애로 멈춘 주문이 있으면 마감 거부됨

## 14. MVP 범위

### 14.1 축제 운영에 반드시 필요 (MVP)
- 메뉴 목록 화면 (분류·가격·품절·이미지/fallback)
- 메뉴 선택 → 장바구니 → 주문하기
- **모든 주문에 학번+이름 입력 필수** (외부인은 체크박스 + 이름만, ADR-021)
- 수령 방법·테이블 번호 입력
- **서버 측 금액 자체 계산 (Pattern B, ADR-020)** — 클라이언트 가격 위변조 차단
- 주문번호 발급
- 주문 완료 화면 (계좌·금액·확인 요청 버튼·조리 현황 보기 버튼)
- 이체 확인 요청 화면 (은행 입력 + 다른 이름 옵션, ADR-021)
- **사용자 조리 현황판** — 학생은 학번+주문번호, 외부인은 토큰+주문번호 (귀여운 카피 + 마스코트, ADR-015)
- 본부 대시보드 (5단계 카드)
- 이체 확인 화면
- 주문 상태 토글
- 메뉴 품절 토글
- 쿠폰 기능 (학번 prefix `202637` 검증 + 1회 사용, ADR-019) — 쿠폰 게이트로 격하, 명부 관리 화면 미도입
- "오늘 정산 마감" 버튼 + 일자별 정산 화면
- **정산 ZIP 백업 다운로드 (개인정보 보호 포함)**
- **자동 ZIP 스냅샷 30분 주기 (ADR-022)** — 정산 ZIP의 안전망의 안전망
- **인기 메뉴 실시간 랭킹 (동적 카피 포함)**
- **Docker 기반 배포 (ADR-023)** — `docker compose up -d` 1줄로 가동, volume 영속화

### 14.2 있으면 좋은 기능 (Phase 2)
- 사장님 추천 자동 인기 계산 정밀화
- 시간대별 주문량 그래프 정밀 시각화
- CSV/PDF 정산 별도 내보내기 (ZIP 안에 이미 포함됨)
- 거부 쿠폰 시도 상세 로그
- 마감 임박 배너
- 알림 음 다양화
- ZIP 백업 패스워드 보호 강화 (사용자 정의 비번)
- 조리 현황판 SSE/WebSocket push 방식 전환

### 14.3 이번 버전 제외
- 결제 시스템(PG)
- 세트 메뉴
- 회원 가입·로그인 (사용자측)
- 푸시 알림
- 멀티 부스 동시 운영
- 다국어
- 다크 모드
- 영수증 인쇄
- 주문 폭주 별도 모드
- **대기 시간 표시 (어떤 방식이든 미도입 — `ADR-010`, D8 재확인)**
- **혼잡도 단계 표시 (D8에서 reject)**
- 본부 음성(TTS) 알림 (D3 skip)
- 부스 입구 대형 호명 모니터 (D4 skip)
- 주문번호 카톡/SNS 공유 (D5 skip)
