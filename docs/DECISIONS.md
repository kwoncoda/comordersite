# Architecture Decision Records (ADR)

이 문서는 **오늘 저녁은 치킨이닭!** 프로젝트의 중요한 설계·운영 결정을 시간 순으로 기록합니다.
형식: 번호 / 제목 / 상태 / 컨텍스트 / 결정 / 결과.

상태 정의:
- **Proposed**: 제안된 상태, 아직 합의 전
- **Accepted**: 수용된 상태, 현재 유효
- **Superseded**: 다른 ADR에 의해 대체됨 (어느 ADR이 대체했는지 명시)

관련 기획 문서: [`order-system-plan.md`](./order-system-plan.md)

## ADR-003: 결제 처리 = 사용자 자가 이체 + 사용자 신고 + 운영진 확인

**상태:** Accepted (2026-04-28)

**컨텍스트:**
- 외부 PG(결제대행) 도입은 학교 축제 규모/책임 구조상 과함
- 단순 "계좌번호 안내"만 두면 입금 매칭이 어려워 운영진 부담이 큼

**결정:**
결제 흐름을 다음 3단계로 운영한다:
1. 사용자가 메뉴 확정 → 상태 `ORDERED` (주문 중)
2. 사용자가 본인 은행앱에서 이체 후 사이트로 돌아와 **"확인 요청" 버튼**을 누르며 **이름과 은행**을 입력 → 상태 `TRANSFER_REPORTED` (이체 확인 요청)
3. 운영진(본부 담당자)이 **통장과 대조**(이름·은행·금액·시각 4가지) 후 "이체 확인" 토글 → 상태 `PAID` (이체 완료) → 조리 시작

매칭 실패 시 보조 상태 `HOLD`(보류). 사용자가 정보 수정 후 재제출 가능.

**결과:**
- 사용자 화면에 "확인 요청" 버튼 + 이름/은행 입력 폼 추가
- 본부 담당자 전용 "이체 확인 화면" 신규 추가
- 노쇼 위험을 0에 가깝게 줄임 (이체 확인 전 조리 안 함)
- 본부 담당자 1명 이상 상시 배치 필요 (병목 지점)

## ADR-004: 1학년 쿠폰 검증 = 명부 매칭 (학번 9자리 + 이름)

**상태:** ~~Accepted (2026-04-28)~~ → **Superseded by ADR-019 (2026-05-03)** — 명부 미보유로 학번 prefix 검증 방식으로 전환

**컨텍스트:**
- "학번 형식만으로 검증"은 다른 학년이 학번을 변조해 시도할 수 있어 약함
- 학생회는 1학년 학번·이름 명부를 사전 확보 가능
- 2026-05-02 로고 확정으로 운영 학과가 **컴퓨터모바일융합과(컴모융)**으로 명시됨
- 2026-05-03 T2 결정: 축제 당일 명부 누락 발견 시 처리 정책 결정

**결정:**
- 학번은 9자리 숫자
- 검증 = **사전 등록된 명부(학번+이름)와의 정확한 매칭**
- 명부 대상 학과 = **컴퓨터모바일융합과 1학년**
- 명부에 없으면 거부, 명부에 있어도 이미 사용한 학번이면 거부
- 카피 톤: "**컴공 새내기 환영 쿠폰**" 또는 "**컴모융 1학년 인증 미션**"
- **명부 갱신 정책 (T2, 2026-05-03):** 축제 당일 추가 등록 불가. 사전 등록 명부만 사용. 누락된 1학년이 있어도 현장에서 등록하지 않음 — 공정성 우선. 운영 가이드에 "명부는 5/20 오픈 24시간 전에 최종 확정" 명시.
- **다일 운영 (G8, 2026-05-03):** 학번당 1회 사용은 축제 전체 공유. 5/20에 사용한 학번은 5/21에 재사용 불가.

**결과:**
- 관리자 페이지에 "명부 관리" 화면 추가 (CRUD + CSV 업로드)
- 쿠폰 활성화의 사전 조건 = "명부 등록 완료"
- 개인정보(학번·이름)는 정산 후 N일(권장 7~14일) 내 자동 삭제
- 운영 가이드에 명부 마감 시점·축제 중 변경 금지 명시 필요

## ADR-006: 메뉴 이미지 = 사진 우선 + 이모지 fallback

**상태:** Accepted (2026-04-28)

**컨텍스트:**
- 메뉴 사진을 모두 준비하지 못할 가능성
- 사진 없는 카드를 빈칸으로 두면 시각적 정돈이 깨짐

**결정:**
- 기본은 사용자가 제공할 사진
- 이미지 없거나 로딩 오류 시 **이모지 + 컬러 배경**으로 fallback
- 분류별 매핑: 치킨 🍗 노란, 사이드 🍟 주황, 음료 🥤 파랑

**결과:**
- 사진 누락 시에도 일관된 시각 경험
- 이미지 비율 통일 (4:3 또는 1:1), 권장 용량 ≤ 200KB

## ADR-007: 주문 폭주 별도 모드 미도입

**상태:** Accepted (2026-04-28)

**컨텍스트:**
- "주문 폭주 모드 토글"은 분기 로직을 늘리고 테스트 부담 ↑

**결정:**
별도 폭주 모드를 도입하지 않는다. 폭주 시 운영 권장사항으로 대응한다.

**결과:**
- 시스템 분기 로직 없음 → 구현/테스트 단순화
- 운영 가이드에 폭주 권장 절차 명시

## ADR-009: 결제 시스템(PG) 미도입

**상태:** Accepted (2026-04-28)

**컨텍스트:**

- 결제 관련 라이브러리/SDK 의존성 0
- PCI-DSS·결제 보안 컴플라이언스 부담 없음
- 환불은 운영진의 수동 계좌이체로만 처리

## ADR-010: 대기 시간 표시 기능 미도입

**상태:** Accepted (2026-04-29) — supersedes ADR-005 · 2026-05-02 D8에서 재확인 (혼잡도 단계 표시도 reject)

**컨텍스트:**
- ADR-005에서 운영진 토글 방식을 채택했으나, 운영자 토글 누락 위험과 시스템/UI 복잡도 부담이 남아 있음
- ADR-014로 마감 시각이 **가변**으로 결정되어, 대기시간을 약속할 근거가 더 약해짐
- 사용자가 "대기시간 기능 자체를 빼고 싶다"고 명시
- **2026-05-02 추가 기획에서 "혼잡도(여유/보통/혼잡) + 시간"을 운영진 수동 토글로 다시 제안 → /plan-ceo-review D8 결정에서 이마저도 reject**

**결정:**
**대기 시간 표시 기능을 어떤 방식으로도 도입하지 않는다.**
- 자동 계산 X
- 운영진 토글 X
- 단순 안내 문구 X
- **혼잡도 단계 표시 X** (D8에서 2026-05-02 재확인)

사용자 기대 관리는 **ADR-015 조리 현황판**이 흡수한다 (시간 약속 없이 "어느 단계에 있는지"만 알려줌).

**결과:**
- 본부 화면 상단의 대기시간 토글 UI 미구현
- 사용자 주문 완료 화면의 "예상 대기시간" 항목 미표시
- 혼잡도 이모지/라벨도 미표시
- §13.8 "대기 시간이 길어질 때" 시나리오는 "마감 시각이 다가올 때"로 재해석
- ADR-005는 Superseded 처리

**Trade-off:**
- 폭주 시 사용자에게 안내할 시간 정보 없음 → 부스 입구 정적 안내 + 조리 현황판이 흡수

## ADR-011: 서비스명 변경 = "오늘 저녁은 치킨이닭!"

**상태:** Accepted (2026-04-29) — supersedes ADR-001 · 2026-05-02 로고 확정

**컨텍스트:**
- ADR-001에서 "컴모융오더"로 정했으나 사용자가 새 이름을 확정
- 치킨 부스의 캐주얼한 분위기와 학생 유머 톤을 살리는 이름이 필요

**결정:**
서비스명을 **"오늘 저녁은 치킨이닭!"**로 확정한다.
- 짧은 별칭: **치킨이닭** 또는 **오저치**
- 마지막 "닭!" (느낌표 포함)을 시각적으로 강조
- **부 카피: "WINNER WINNER CHICKEN DINNER!"** (PUBG 승리 메시지 차용 — 일반 카지노 슬랭이라 사용 가능, ADR-013 가드레일 적용)
- **마스코트: 자체 제작 군용 헬멧 곰 캐릭터** (`웹 로고.png`, 2026-05-02 사용자 확정)
- 운영 주체 표기: **컴퓨터모바일융합과**

**결과:**
- 모든 UI 헤더, 페이지 타이틀, 메타 태그, OG 이미지 텍스트가 새 이름으로 통일
- "WINNER WINNER CHICKEN DINNER!" 부 카피는 주문 완료 화면·완료 단계·로고에 활용
- 마스코트 곰은 메뉴 fallback·조리 현황판 단계 표시·빈 상태 등에 적극 활용
- 로고 파일 경로: `웹 로고.png` (프로젝트 루트, MVP 단계에서 `public/` 또는 `static/`으로 이동 예정)
- ADR-001은 Superseded 처리

## ADR-012: 정산 트리거 = 관리자 수동 "오늘 정산 마감" 버튼

**상태:** Accepted (2026-04-29) · 2026-05-03 진행 중 가드 정책 추가 (강제 마감 제거)

**컨텍스트:**
- 축제 마감 시각이 **가변** (ADR-014). 자동 마감 시각을 시스템에 박을 수 없음
- 5/20과 5/21 두 일자 운영 → 일자별 정산 분리 필요
- 관리자가 현장 분위기를 보고 마감을 결정하는 게 더 현실적
- 2026-05-03 T3 결정: "강제 마감 + 마감 후 보류" 분기가 모호해 운영 혼란 우려

**결정:**
정산은 **관리자가 정산 화면의 "오늘 정산 마감" 버튼을 직접 눌러** 트리거한다.
- 버튼 클릭 시 그 시점까지의 데이터로 일자별 정산 스냅샷 생성
- **진행 중 주문 가드 (T3, 2026-05-03):** `ORDERED`/`TRANSFER_REPORTED`/`PAID`/`COOKING`/`READY`/`HOLD` 상태가 1건이라도 있으면 **마감 거부**. 팝업으로 안내:
  > "진행 중 주문 N건이 있습니다. 모든 주문이 완료(또는 취소)된 후 다시 시도해 주세요."
- **강제 마감 미도입.** "마감 후 보류" 상태도 도입하지 않는다 — 개념 자체 제거.
- 모든 주문이 `DONE` 또는 `CANCELED` 상태일 때만 마감 진행 가능
- 일자별 정산 + 합산 정산 화면 모두 제공

**결과:**
- 정산 자동화/스케줄러 로직 불필요
- 5/20과 5/21을 별개 정산으로 보관
- 합산 화면은 두 정산 결과를 단순히 더해서 표시
- 운영 체크리스트에 "정산 마감 버튼 클릭"을 필수 항목으로 명시
- 운영 가이드에 "마감 시각 도달 → 새 주문 차단 → 진행 주문 모두 처리될 때까지 대기 → 마감 버튼" 흐름 명시
- **데이터 일관성 보장:** 정산 스냅샷 이후 주문 상태가 변경되는 일이 없음 (모두 `DONE`/`CANCELED`로 종결)

**Trade-off:**
- 마감 시각 결정의 자유도 감소 (모든 주문이 끝나야 함)
- 운영진이 마감 직전 새 주문 차단을 깜빡하면 마감이 늦어질 수 있음 → 운영 가이드로 보강

## ADR-013: 디자인 컨셉 = 배달의민족 UX + PUBG 비주얼 차용

## ADR-015: 사용자 조리 현황판(실시간) MVP 도입

**상태:** Accepted (2026-05-02) · 2026-05-03 실시간 메커니즘 = SSE 결정 · 2026-05-04 인증 정책 확정 (D17 closed, ADR-021과 결합)

**컨텍스트:**
- ADR-010으로 대기시간 표시를 통으로 제거했으나, 사용자의 "조리 완료됐나?" 불안은 그대로 남아 부스 문의 폭증 우려
- 시간을 약속하지 않으면서도 "어디까지 진행됐는지"를 알려주는 대안이 필요
- 단순 텍스트 진행 표시로는 학생 축제 톤·재미 부족

**결정:**
사용자가 주문번호로 자기 주문의 현재 단계를 폰에서 직접 확인하는 **조리 현황판**을 MVP에 도입한다.

**핵심 명세:**
- 7개 시스템 상태 → **귀여운 카피 + 마스코트 단계 표시**로 변환
  - 예: `COOKING` → "🔥 지금 치킨이 기름 속으로 입장했습니다!"
  - 예: `READY` → "✅ #17번, 수령 가능해요!"
- 5단계 progress bar (접수 → 입금확인 → 조리 → 마무리 → 수령)
- 마스코트 곰이 단계별 변형으로 등장
- **실시간 메커니즘 = SSE (Server-Sent Events)** (1A 결정, 2026-05-03): 서버가 상태 변경 시 즉시 push, 클라이언트 폴링 X
  - HTTP 기반이라 프록시·방화벽 친화적
  - 클라이언트는 `EventSource` API로 단순 구현
  - 동시 연결 30개 이하라 서버 부담 무시 수준
  - 연결 끊어지면 자동 재연결 (브라우저 기본)
- 시간 약속 X (어떤 분 단위 추정도 표시 안 함)

**인증 정책 (D17 closed, 2026-05-04):**
- ADR-021로 모든 주문에 학번+이름이 필수 입력되므로, 조리 현황판 인증은 **학번+주문번호 매칭**으로 단순화한다
  - 학생: 학번(9자리) + 주문번호로 본인 주문만 조회
  - 외부인 (학번 미보유 체크): 토큰(주문 시 발급, URL+쿠키 저장) + 주문번호로 조회. 학번 없는 외부인은 동명이인 충돌 위험이 있어 토큰 fallback 필요.
- 폰번호 뒷자리 보조 키 미도입 (PII 추가 수집 부담만 있고 보안 추가 효과 미미)
- 쿠키 만료 24시간 권장 (`/plan-eng-review`에서 확정)

**결과:**
- 신규 사용자 화면 1개 (조리 현황판)
- 주문 완료 화면에 "조리 현황 보기" 버튼 추가
- 주문번호 + 토큰 발급 로직 + 상태 변경 자동 반영 폴링 API 추가
- ADR-010과 상충하지 않음 — 시간을 약속하지 않으므로 운영진 토글 누락 위험 0
- 부스 문의 "제 주문 언제 나와요?" 흡수 효과 기대

**Trade-off:**
- SSE 연결 30개 동시 유지 — 서버 메모리·소켓 부담 모니터링 필요
- 일부 구버전 브라우저는 SSE 미지원 — 학교 축제 사용자는 모바일 최신 브라우저 가정이라 무시 가능
- Phase 2에서 WebSocket으로 확장 (대시보드 실시간 칩 등 추가 시)

## ADR-019: 1학년 쿠폰 검증 = 학번 prefix 검증 (명부 미보유 우회)

**상태:** Accepted (2026-05-03) — supersedes ADR-004

**컨텍스트:**
- ADR-004는 사전 등록 명부와의 매칭을 전제로 했으나, 사용자가 명부를 확보할 수 없는 상황 발생
- 컴퓨터모바일융합과 1학년 학번은 **앞 6자리가 `202637`로 고정**되어 있고, 마지막 3자리만 학생별로 다름 (`202637XXX`)
- 학번 형식 자체가 1학년 식별에 충분한 정보 — 명부 매칭 없이도 검증 가능

**결정:**
1학년 쿠폰 검증은 **학번 prefix 검증**으로 전환한다. 명부는 도입하지 않는다.

**검증 로직:**
```
1. 학번이 9자리 숫자인지 (정규식: ^\d{9}$)
2. 학번이 "202637"로 시작하는지 (substring 6자)
3. 이름이 한글 1자 이상인지 (비어 있지 않음, 최대 길이 제한)
4. 같은 학번이 이미 쿠폰을 사용한 적 있는지 (used_coupons 테이블 unique key 검증)
모두 통과 → 1,000원 할인 적용
```

**사용자 입력 (변경 없음):**
- 학번 9자리
- 이름 (한글)

**이름의 역할 (변경):**
- 검증용이 아닌 **식별·기록 용**: 운영진이 사후 학생증과 대조 / 정산 회의 시 누가 사용했는지 기록
- 이름 형식 검증은 "비어있지 않음" 수준만

**같은 학번 1회 제한:**
- ADR-018(T1)에 따라 축제 전체 기간(5/20+5/21) 학번당 1회로 유지
- used_coupons 테이블이 명부 역할 대체

**결과:**
- "명부 관리" 화면·CSV 업로드 기능 미도입 (관리자 페이지에서 제거)
- 운영진의 사전 명부 확정 작업 불필요
- 쿠폰 사용 즉시 used_coupons 레코드 생성: (학번, 이름, 사용시각, 주문번호, 할인액)
- 시도 거부 로그는 사유별로: prefix 불일치 / 형식 오류 / 중복 사용
- 정산 후 N일(권장 7~14일) 내 used_coupons 자동 삭제 정책 유지

**Trade-off (위조 위험 인지):**
- ⚠️ 다른 학년이 `202637`로 시작하는 임의 9자리 숫자 + 임의 이름을 입력하면 검증 통과
- ✅ 완화 요인:
  - 같은 학번 1회 제한 → 같은 위조 학번 반복 사용 불가
  - 1,000원 할인 수준이라 위조 incentive 제한적
  - 사용자가 학번/이름을 적어 사후 운영진이 학생증 대조 가능 (명목적 억제력)
- 더 강한 검증이 필요해지면 Phase 2에서 학번 마지막 3자리 범위 검증 등 추가 가능

**Supersession 영향:**
- ADR-018의 명부 갱신 정책(T2: "당일 추가 등록 불가")은 명부 자체가 없으므로 무의미 → 삭제
- ADR-018의 학번 unique 정책(T1)은 그대로 유지 (used_coupons unique key)

**결정:**
- **주문번호:** 일자별 리셋. 5/20 16:30 #1부터, 5/21 16:30 #1부터. 일자 + 번호로 unique 식별.
- **학번 unique key:** 축제 전체 공유. 5/20에 쿠폰 사용한 학번은 5/21에 재사용 불가 (ADR-004의 "학번당 1회"가 축제 전체 기준).
- **명부:** 사전 등록된 명부는 양일 공유 (ADR-004 참고).
- **조리 현황판 세션:** 일자별. 5/20 주문은 5/20 마감 후 사용자에게 더 이상 노출 안 됨 (사용자 시점 단순화).
- **관리자 대시보드:** 일자 필터 (오늘/어제) + 전체 합산 보기 둘 다 지원.

**결과:**
- 주문번호 식별자는 `(일자, 번호)` 복합 키 또는 `날짜-번호` 단일 문자열
- 사용자 화면의 "#17/100" 도그태그는 일자별 리셋이라 양일 모두 작은 숫자부터 시작
- 학번 unique 인덱스는 (학번) 단일 — 일자 컬럼 추가 안 함
- 정산 화면에 "5/20 / 5/21 / 전체 합산" 3가지 뷰 제공

**Trade-off:**
- 주문번호 리셋과 학번 공유의 의미 경계가 직관적이지 않을 수 있음 → 운영 가이드에 명시
- 사용자가 5/20 주문 이력을 5/21에 조회 못 함 → 단순한 사용자 경험 우선

## ADR-020: 주문 금액은 서버가 메뉴 ID·수량·쿠폰만 받아 자체 계산 (Pattern B)

**상태:** Accepted (2026-05-04)

**컨텍스트:**
- 클라이언트가 보낸 합계·가격을 서버가 신뢰하면 DevTools 변조로 가격 위변조 가능
- 운영진의 통장 매칭(§6.4)은 "통장 입금↔시스템 표시 금액" 일치만 검증하므로, 시스템 표시 금액 자체가 변조되어 있으면 그대로 통과 (예: 사용자가 100원으로 변조 → 100원 이체 → 100원=100원 매칭 통과 → 18,000원어치 음식 수령)
- 컴퓨터모바일융합과 학생 부스 특수성: 일반 부스보다 변조 시도 확률 높을 가능성

**결정:**
주문 API는 **클라이언트가 보낸 가격·합계·할인액을 신뢰하지 않는다.** 서버는 다음 항목만 받아 자체 계산한다:
- `menu_id` 목록과 각 항목 수량
- 쿠폰 코드 (적용 시)
- 학번·이름 (ADR-021)
- 수령 방법·테이블 번호

**서버 계산 흐름:**
```
1. 각 menu_id의 가격을 서버 DB에서 lookup
2. 수량과 곱해 항목별 소계 계산
3. 쿠폰 검증 (ADR-019: 학번 prefix + used_coupons unique)
4. 쿠폰 적용 시 할인 적용 (ADR-019의 1,000원)
5. 최종 합계 = 서버 계산값으로만 결정
6. orders 테이블에는 서버 계산값만 저장
```

**클라이언트 측 가격 표시:**
- 사용자 화면의 "21,000원" 등은 *표시용*이며, 서버 결제 금액과 다를 가능성이 0에 가깝지만 불일치 시 서버 값이 권위
- 메뉴 가격이 운영 중 변경되는 경우(품절/단종은 토글, 가격 변경 시나리오는 미상정)는 트랜잭션 시작 시점의 DB 값 사용

**결과:**
- 위변조 시도가 *원천적으로 불가* — 클라이언트는 가격 정보를 서버에 보낼 채널이 없음
- 구현 시 자연스러운 패턴이지만, ADR로 못 박아 구현·테스트 단계에서 누락 0%
- 테스트 케이스 필수 (4개): ① 정상 주문 ② 클라이언트가 다른 total을 보내도 거부/무시 ③ 존재하지 않는 menu_id 거부 ④ 쿠폰 위조(다른 학년 학번 prefix) 거부
- OWASP Top 10 A04 (Insecure Design) 기본 방어

**Trade-off:**
- 추가 코드 부담 거의 없음 (자연스러운 구현 패턴)
- 클라이언트 측 표시값과 서버 값이 *극히 드물게* 일치하지 않을 가능성 (예: 메뉴 단가 운영 중 수정 — 운영 가이드에 "운영 중 메뉴 가격 변경 금지" 명시 권장)

## ADR-021: 모든 주문에 학번+이름 필수 입력 (외부인은 "학번 없음" 체크박스)

**상태:** Accepted (2026-05-04) — supersedes 부분: ADR-019의 학번 prefix 검증 적용 범위 (전체 주문 → 쿠폰 게이트 한정)

**컨텍스트:**
- ADR-019는 학번/이름 입력을 "쿠폰 사용 시에만"으로 가정했으나, 이 모델은 D17(조리 현황판 인증)을 별도 토큰·폰번호로 풀어야 하는 부담 발생
- 학교 축제 주 고객층은 학생이라 학번 입력 마찰이 일반 폼 입력 수준 (외움)
- 학번+이름이 모든 주문에 있으면:
  - 조리 현황판 인증이 자연 해소 (ADR-015)
  - 본부 이체 매칭 단순화 (이름이 주문 시점에 이미 있음)
  - 도그태그 SNS 공유 시 타인 사칭 위험 ↓ (학번을 모르면 조회 불가)
- 외부인(학부모·동생·졸업생)도 일부 존재하므로 완전 차단은 매출 손실

**결정:**
모든 주문에 다음 입력을 **필수**로 받는다:
- **학번** (9자리 숫자) + **이름** (한글 1자 이상)
- 또는 **"학번 없음" 체크박스** + 이름만

**입력 흐름:**
```
주문 폼:
  □ 학번 없음 (외부인)
  학번: [202637XXX]   ← 체크박스 미체크 시 필수, 9자리
  이름: [홍길동]       ← 항상 필수
```

**검증:**
- 학번이 입력된 경우: 9자리 숫자 형식 검증
- 학번 prefix `202637` 검증은 **쿠폰 적용 시에만** (ADR-019 격하): 1학년 컴퓨터모바일융합과 식별 게이트
- 외부인(체크박스): 학번 검증 skip, 이름만 검증 (1자 이상)

**조리 현황판 인증 (ADR-015 보강):**
- 학생: `학번 + 주문번호` 매칭으로 본인 주문 조회
- 외부인: `토큰(쿠키+URL) + 주문번호` 매칭 (학번 없으므로 토큰 fallback)

**이체 확인 매칭 (§6.4 보강):**
- 주문자 이름이 주문 시점에 이미 입력되어 있음
- 이체 확인 화면에선 "이체자 이름 = 주문자 이름" 기본 가정, "다른 이름으로 이체했나요?" 옵션 체크박스로 처리 (가족 대리 이체 등)
- 본부 매칭이 4요소(이름·은행·금액·시각) 그대로 유지 (단 이름은 시스템에 이미 있음)

**PII 영향:**
- 모든 주문에 학번·이름 보관 (이전: 쿠폰 사용분만)
- 정산 후 N일(권장 7~14일) 자동 삭제 정책 — 적용 범위 확대 (orders 테이블 + used_coupons 테이블)
- 외부인(학번 없음)도 이름은 동일 정책 적용
- 운영 가이드·약관에 "학번·이름 수집 목적 = 주문 식별·이체 매칭·조리 현황판 인증·정산 후 삭제" 명시

**결과:**
- §4.1 step 8 갱신: "쿠폰 사용 여부" 분기 제거, 학번+이름은 항상 필수
- §10.2 이체 확인 화면 갱신: "이체하신 이름" 입력란 → "이체자 이름 = 주문자 이름과 동일" 체크박스 + 다른 이름 옵션
- §10.4 조리 현황판 인증 흐름 갱신: 학번/외부 토큰 분기
- ADR-019의 학번 prefix 검증은 쿠폰 게이트로 격하 (전체 주문 게이트가 아님)
- D17 closed (별도 토큰·폰번호 보조 키 미도입)

**Trade-off:**
- 모든 사용자 주문 마찰 +10~15초 (학번 9자리 + 이름)
- PII 보관 범위가 모든 주문으로 확대 (운영 가이드·삭제 정책 강화 필요)
- 외부인 처리 복잡성 (체크박스 분기) — 단 외부인 주문 비율 낮을 것으로 예상
- 동명이인 외부인 처리 한계 (토큰 의존)

## ADR-022: 자동 ZIP 스냅샷 30분 주기 (정산 ZIP의 안전망의 안전망)

**상태:** Accepted (2026-05-04)

**컨텍스트:**
- ADR-016 ZIP 백업은 *정산 마감 시점*에만 생성됨
- 운영 중(예: 5/20 19시) DB 손상·디스크 이슈·서버 강제 종료 시 데이터 일부 손실 → 정산 시점 ZIP은 *불완전*
- ADR-023의 Docker volume 모델로 디스크 쓰기·장시간 가동이 안정 확보됨 → 자동 스냅샷이 자연스러움

**결정:**
서버는 운영 중 **30분 주기로 자동 ZIP 스냅샷**을 생성하여 마운트된 Docker volume의 `backups/` 디렉터리에 저장한다.

**스냅샷 구조 (수동 ZIP과 동일):**
- `manifest.json`, `orders.json`/`csv`, `coupons.json`/`csv`, `menu-snapshot.json`, `images/`, `README.txt`
- 단 `settlement-summary.pdf`는 자동 스냅샷에선 생략 (정산 마감 전이라 의미 없음)

**회전 정책:**
- 누적 6개만 유지 (3시간치)
- 가장 오래된 스냅샷 자동 삭제
- 디스크 차지: 평균 100KB-1MB × 6 = 6MB max (무시 수준)

**구현 방식:**
- 서버 프로세스 내부 `setInterval` 또는 cron (Node 단일 프로세스 가정)
- 수동 정산 ZIP 생성 로직과 동일 함수 재사용 (DRY)
- 자동 스냅샷은 다운로드 이력에 *자동* 마커로 표시 (수동/자동 구분)

**보안·개인정보:**
- ADR-016과 동일 정책 (학번·이름·은행 정보 포함)
- Docker volume 디렉터리 권한 관리
- 정산 후 N일 폐기 시 자동 스냅샷도 함께 삭제

**결과:**
- 데이터 손실 최악값을 30분으로 제한
- 사고 시 운영 가이드: "최신 자동 스냅샷을 USB 복사 후 학생회 보관"
- 정산 화면에 자동/수동 스냅샷 이력 모두 표시
- 구현 부담 ~30줄 (타이머 + 회전 + 기존 ZIP 함수 재사용)

**Trade-off:**
- 30분 주기는 절충값 — 5분이면 디스크 I/O 부담↑·1시간이면 손실 윈도우↑
- 자동 ZIP에도 PII 포함 → Docker volume 보안 + 사후 폐기 정책 엄수
- 운영진이 자동 스냅샷의 존재를 모를 시 USB 복사 누락 가능 — 운영 가이드에 명시

## ADR-023: 배포 = Docker 기반 (Docker volume 영속화)

**상태:** Accepted (2026-05-04)

**컨텍스트:**
- 호스팅 환경 결정 미루면 ADR-022 자동 스냅샷·§13.11 수동 재시작 부담·§12.5 ZIP 다운로드 등 다수 결정이 진공 상태
- 학교 축제 운영 환경은 호스트 다양성이 높음 (학생회 노트북 / VPS / 친구 컴퓨터 / 학교 서버) → 이식성이 가치
- 사용자가 Docker로 결정

**결정:**
배포 모델은 **Docker 컨테이너 + Docker volume** 모델로 통일한다.

**아키텍처:**
```
[호스트 머신 (학생회 노트북 / VPS / 학교 서버)]
  └── Docker daemon
        ├── 컨테이너: chickenedak-server (Node 앱)
        │     └── 마운트: /app/data → docker volume
        └── Volume: chickenedak-data
              ├── db.sqlite (또는 사용 DB)
              ├── backups/ (자동 + 수동 ZIP 스냅샷)
              ├── images/ (메뉴 사진)
              └── logs/
```

**구성 요건:**
- 호스트 = Docker daemon만 설치되어 있으면 됨 (Linux/macOS/Windows 모두)
- `docker compose up -d` 1줄로 가동
- `restart: always` 정책 (컨테이너 죽어도 자동 부활)
- Volume은 호스트 디렉터리 또는 named volume — 백업 = `docker run --rm -v chickenedak-data:/data alpine tar czf /backup.tar.gz /data` 1줄

**운영 절차 (§13.11 보강):**
- 재시작 = `docker compose restart` (5초)
- 로그 확인 = `docker compose logs -f`
- 컨테이너 상태 = `docker compose ps`
- 백업 = volume tar로 USB 복사 (자동 ZIP과 별개의 disaster recovery)

**결과:**
- 호스팅 이식성 확보 (어디든 가동)
- 자동 재시작으로 §13.11 수동 재시작 부담 감소
- ADR-022 자동 스냅샷이 안정적으로 작동할 환경 확보
- 운영진의 도커 학습 부담: 명령 4-5개 (`up`, `down`, `restart`, `logs`, `ps`) — D-1 리허설 시 5분 설명으로 충분
- 인수인계 자산화 (Dream State): "이 시스템은 Docker로 돌아간다" 1줄로 전달

**Trade-off:**
- 운영진이 Docker 명령에 익숙해야 함 (vs `npm start` 같은 가벼운 명령)
- 호스트가 Docker를 못 쓰는 환경(예: 학교 정책으로 root 권한 없는 공용 PC)이면 배포 불가 → 사전에 호스트 환경 확인 필요
- Docker 설치 자체가 호스트의 첫 설정 단계 (학생회 노트북에 사전 설치)
- 이미지 빌드 시간 (CI 없이 로컬 빌드면 ~1-3분) — 자주 배포는 권장 안 함

## ADR-024: 기술 스택 — Node 20 + Express + SQLite + EJS + Alpine.js

**컨텍스트:**

**결정:**

| Layer | 선택 | 근거 (한 줄) |
|---|---|---|
| Runtime | Node.js 20 LTS | Docker 어디서든 안정, ESM 호환 |
| Language | JavaScript + JSDoc | TS 빌드 비용 > 타입 버그 절감 (16일 일정) |
| HTTP framework | Express 4.x | 문서·답변 가장 풍부, 학생 부스 트래픽엔 차이 무의미 |
| DB | SQLite (better-sqlite3) | 30 동시엔 SQLite WAL이 더 빠르고 백업 단순 |
| 마이그레이션 | raw SQL 파일 + `_migrations` 추적 | ORM 학습 비용 과함 |
| 템플릿 | EJS 3.x | 서버 렌더 단순, 빌드 단계 0 |
| 클라이언트 인터랙션 | Alpine.js 3.x (CDN) | SPA-lite, 빌드 X, 11KB |
| CSS | Tailwind CSS 3.x (CLI) | PUBG 톤 유틸 클래스, JIT 출력 작음 |
| 검증 | zod 3.x | 입력/출력 모두 활용, Pattern B 가드 |
| 로깅 | pino 9.x | 구조화 JSON, Docker logs와 잘 맞음 |
| ZIP | archiver 7.x | Node 표준급, 스트리밍 |
| 세션 (관리자) | express-session + connect-sqlite3 | 단일 서버 + 쿠키 세션이면 충분 |
| 보안 헤더 | helmet 8.x | 기본 보안 헤더 자동 |
| 환경변수 | dotenv 16.x | `.env` + Docker 호환 |

**결과:**
- `package.json` 확정 dependencies 가능
- 빌드 단계 = Tailwind CLI 1개만 (프론트엔드 빌드 X)
- `node src/server.js` 1줄로 가동 (Docker 안에서)
- TypeScript는 1차 운영 후 마이그레이션 가능 (Phase 2)
- React/Postgres 등 "Phase 2 기술 부채"가 아니라 "Phase 2에서 필요해지면 도입"하는 의식적 deferral

**Trade-off:**
- TS 미채택으로 IDE 자동완성 강도 ↓ — JSDoc + zod로 부분 보완
- SQLite로 결정해 외부 호스팅·다중 인스턴스 시 마이그레이션 부담 — 학교 축제 단일 부스는 무관
- React 미채택으로 컴포넌트 재사용 패턴 ↓ — 화면 12개·복잡 상태 없으니 over-engineering 회피
- Express 4.x는 "boring"한 선택 — 5.x도 안정화됐지만 변경 사항이 학습 비용
- Alpine.js는 React에 비해 학습 커뮤니티 작음 — 단순 API라 부담 적음

## ADR-026: 디자인 시스템 — Battle Royale Cafeteria + 도그태그 메모러블

**상태:** Accepted (2026-05-07)

**컨텍스트:**
- ADR-013에서 컨셉(배민 UX + PUBG 비주얼 + 자체 마스코트)이 결정되었으나, 구체 토큰(컬러·폰트·간격·모션)·컴포넌트 변형·AI 슬롭 회피 원칙은 미정
- ADR-011 마스코트 + WINNER WINNER + 도그태그 메타포 결정으로 시각 정체성의 *방향*은 잡혔지만, *세부*가 빠짐
- D-day 13일 — 시각 결정이 빠르게 굳어야 시안·구현 단계 진행 가능
- 솔로 개발자라 *시각 일관성*을 토큰 기반 시스템으로 보장 필요

**결정:**

**1. Memorable thing — "주문번호를 받았을 때 도그태그를 받은 느낌"**
- 모든 시각 결정은 이 한 가지를 강화하거나 무관해야 함. 약화하는 결정은 거부.
- 도그태그 = 핵심 컴포넌트 (`DogTagFrame`), 메인 모션(주문 완료 시 떨어짐), 형광 옐로 활용

**2. 컬러 — 라이트 default + 따뜻한 카키 + 형광 옐로 포인트** (2026-05-10 라이트 default로 정정)
- 베이스 라이트 (default): `#F5EFE0` (따뜻한 크림) + `#EAE3D0` (표면) + `#2A2C20` (잉크)
- 베이스 다크 (prefers-color-scheme 보조): `#1A1A14` (짙은 카키) + `#25241A` (표면) + `#F5EFE0` (잉크)
- 포인트: `#F4D200` (형광 옐로) — CTA·도그태그·READY 알림
- 카모 액센트: 올리브·사막·흙갈색 — *5-10% opacity* 텍스처에만 (단색 X)
- 시맨틱: success #5A8C42 / warning #E59B0C / danger #C73E1D / info #3A6B7E
- 도장 컬러: 빨강 #B5301A, 검정 #1F1B14
- WCAG AA 4.5:1 이상 보장 (잉크/배경 14:1)

**3. 타이포그래피 — Pretendard + JetBrains Mono + Black Ops One**
- 한글 본문/디스플레이: Pretendard Variable (한국 웹 표준, 무료, *system-ui와 Inter 거부*)
- 도그태그·금액·학번: JetBrains Mono (tabular-nums, 양각 새김 느낌)
- "WINNER WINNER" 등 영문 강조: Black Ops One (Google Fonts, 1-2 곳만)
- 스케일: 10/12/14/16/18/20/24/32/48/64px
- 굵기: 400/500/600/700/900 (Pretendard Variable)

**4. 간격·라디우스·모션**
- Spacing: 4px base, 8-step (2/4/8/12/16/24/32/48/64)
- Border radius: 8px default, 24px+ 금지 (둥글둥글 bubble = 슬롭), 0px는 도장·사진에만
- Motion: 펀치한 짧은 액션 (100-200ms), 도그태그 떨어짐만 길게(600ms) + 단발성
- 그림자: blur 작은 *오프셋 강조*형 (도장 톤). 부드러운 `shadow-md` 거부

**5. 마스코트 사용 규칙**
- 5종 변형: 기본·출동 준비·조리 중·도착·취소
- 사용 위치: 빈 슬롯·조리 현황판 단계·에러 화면만
- *운영진 화면 사용 금지* (정보 노이즈)
- 카피는 짧음·액션 중심·농담 OK

**6. 농담의 위치**
- 사용자 *돈·결제·에러* = 진지 (농담 X)
- 사용자 *기다림·축하·조리 단계* = 농담 OK (마스코트·"WINNER WINNER")
- *운영진 화면 전체* = 진지

**7. AI 디자인 슬롭 회피 (이 프로젝트 특화 20개)**
- 대표: 보라/바이올렛 그라데이션, 24px+ bubble radius, system-ui/Inter 본문, 진짜 위장 무늬, 3-column 아이콘 그리드, 그라데이션 CTA, 중앙 정렬 모든 것, 라이트/다크 토글 버튼, 영어 마케팅 카피, 큰 부드러운 그림자
- 자체 검증 질문: "이 화면을 진짜 디자이너가 자기 이름 걸고 보여줄 수 있나?"

**8. 두 가지 deliberate risks**
- Risk 1 (도장): "RECOMMENDED"·"SOLD OUT"·"PAID" 등을 *진짜 회전·blur 외곽 도장*으로 (clean SaaS chip 거부) — 학교 축제 손맛
- Risk 2 (도그태그): 단순 카드 X, 진짜 도그태그 모양(구멍·양각·모노스페이스) + 떨어지는 모션 (600ms, 단발) — Memorable thing 강화

**9. 접근성 (WCAG AA + 모바일·축제 환경 보강)**
- 본문 대비 4.5:1, 큰 텍스트 3:1
- 모바일 터치 타겟 ≥ 44px, 사용자 메인 CTA ≥ 56px
- prefers-reduced-motion 지원
- Vibration API (READY 알림)
- 5가지 시나리오 직접 검증 (스크린리더·키보드·저시력·시끄러움·한 손)

**결과:**
- 4개 디자인 문서 신규 작성: `DESIGN.md`(시각 토큰), `UX_STRATEGY.md`(원칙·여정), `SCREEN_STRUCTURE.md`(화면 우선순위), `COMPONENT_GUIDE.md`(컴포넌트 명세)
- 시안·구현 단계가 *토큰 기반*으로 일관성 보장
- 인수인계 자산: 내년 학생회가 *왜* 이렇게 만들었는지 즉시 파악 가능 (Dream State)
- D-day 13일 안에 시각 일관성 확보

**Trade-off:**
- 마스코트 5종 SVG·도장 5종 SVG·카모 패턴 등 자산 제작 부담 (디자이너 또는 외주 필요) — 1차 운영 전 자산 미완 시 임시 fallback (이모지·기본 도형)
- Pretendard·JetBrains Mono·Black Ops One = 외부 폰트 3개 로딩 — Google Fonts CDN으로 빠르지만 첫 방문 시 ~50KB 추가
- Risk 2 도그태그 모션은 *한 번만 좋은* 효과 — 사용자가 새로고침하면 매번 등장 → 스토리지로 *세션당 1회만* 표시 권장
- Pattern B + 다크 default + 카모 텍스처 등 *세밀한 부분*이 시안 단계에서 노이즈로 인식될 수 있음 — 시안 검토 시 토큰 검증 필수
