### 1.1 페르소나 ① — 학생 사용자 (주 고객층, 80%+)

**이름:** 박서연, 21세, 컴퓨터모바일융합과 1학년
**상황:** 5/20 17:30, 친구 3명과 부스 앞 줄 서있는 동안 폰으로 주문
**디바이스:** 갤럭시 S22 (안드로이드 14, Chrome), LTE
**기술 친숙도:** 카카오·인스타·배민 일상 사용. F12나 DevTools는 모름.
**축제 감정:** 들떠있음. 시끄러움. 친구와 대화하면서 한 손으로 폰 조작.
**돈에 대한 태도:** 학생 예산. 1,000원 쿠폰이 *작지만 의미 있음*. 학번 prefix 검증 = 자기가 *진짜 1학년인지 확인받는* 의식.
**기대:** 배민 같은 빠른 흐름. 종이 주문보다 *빠르고 깔끔*하기를.
**불안:** "내 주문 잘 들어갔나?" "조리 완료됐나?" — 부스에 직접 묻기 부끄러움.

## 3. 사용자 감정 곡선 (학생 페르소나 기준)

```
감정
긍정  ▲
  +5  │              ⭐                                              ⭐
      │            도그태그                                       수령
  +4  │           발급 받음                                     "받았다!"
      │           ★                                                │
  +3  │      ★    │                              ★              │
      │   주문   │                            "조리                │
  +2  │   접수   │                           시작!"              │
      │   완료   │                            (마스코트            │
  +1  │   ───   │                            출동)               │
   0  ├──┼──────┼─────────────────────────────┼───┼─────────────┼──▶  시간
  -1  │  메뉴   │                                              │
      │  탐색   │     ⚠️ 이체 후 대기            ⚠️             │
  -2  │  (갈등)         "확인 됐나?"          "기다린지           │
      │                                       오래됐는데?"        │
  -3  │                  ⚠️ 조리 중                              │
      │                  "지금 어디?"                            │
부정  ▼                  (조리 현황판이                           │
                          이걸 흡수)                             │
```

**구간별 위험:**
- **메뉴 탐색** (-1): 친구와 갈등 가능 ("뭐 시킬까?") — UX는 *빠른 결정 도움* (인기 랭킹·추천)
- **이체 후~확인 전** (-2 → -1): 가장 큰 불안 구간 — *확인 요청* 버튼이 눈에 띄게 + *상태 변경 즉시 피드백*
- **조리 중** (0~-2): 시간 약속 X (ADR-010) → 조리 현황판이 *카피로 안심* 흡수
- **수령 직전** (+5): 호명 → 도그태그 *형광 옐로 강조* + 주문번호 큰 글씨

## 4. 핵심 UX 원칙 10개

각 원칙은 *기획서·ADR과 어떻게 연결되는지* 명시.

### UX-1: 한 손 모바일 우선 (One-Handed Mobile First)

**원칙:** 사용자 화면의 모든 주요 액션은 *오른손 엄지 1개로 도달 가능*해야 함. CTA는 화면 하단(safe-area 위), 좌측 정렬.
**근거:** ADR-008 (모바일 전용) + 페르소나 ① (한 손 조작) + 페르소나 ② (한 손)
**구현:**
- 메인 CTA 위치 = 화면 하단 sticky bar
- 모달은 bottom sheet (top dialog 금지)
- 입력 필드 폼은 *세로 1열* (가로 2열 금지)

### UX-2: 정보 위계는 단순함 = 큼 (Hierarchy by Size)

**원칙:** 가장 중요한 정보가 *가장 큼*. 색·아이콘·테두리는 *2차*.
**근거:** 부스 환경 (어두움·바쁨) → 한눈에 인식
**구현:**
- 도그태그 주문번호 = 48px+ Black weight
- 메뉴 가격 = 메뉴명만큼 큼 (둘 다 18px+)
- 부 정보 (분류·캡션) = 14px

### UX-3: 진지함과 농담의 분리 (Serious Money, Playful Time)

**원칙:** 사용자의 *돈·결제·에러*는 진지하게. *기다림·축하·완료*는 농담 OK.
**근거:** 페르소나 ① (학생 예산) + 운영진 신뢰
**구현:**
- 결제 금액 화면 = 도장·마스코트 X, 깔끔한 숫자
- 조리 현황판 = 마스코트 + "🔥 입장!" 농담 카피
- 에러 메시지 = "임무 실패!" 같은 농담 X — *사실 + 회복 경로*

### UX-4: 시간 약속 안 함 (No Time Promises, Just Stage)

**원칙:** "5분 남음" "10분 후 도착" 같은 시간 정보 0. *현재 단계*만 표시.
**근거:** ADR-010 (대기시간 미도입)
**구현:**
- 조리 현황판 = 5단계 progress bar + 카피 (분 표시 X)
- 본부 대시보드 = "elapsed_minutes"는 *경고*용 (5분/10분), 사용자 표시 X

### UX-5: 사용자 통제권 보호 (User in Control)

**원칙:** 사용자의 *뒤로가기·새로고침·종료*가 데이터 손실로 이어지면 안 됨. 자동 redirect·자동 modal·자동 carousel 금지.
**근거:** 페르소나 ① (한 손, 실수 가능)
**구현:**
- 주문 완료 화면 *URL 북마크 가능* — 폰 닫고 다시 와도 도그태그·조리 현황 보임
- SSE 자동 재연결 (사용자 행동 X)
- 모달은 X 버튼 + 외부 클릭 모두로 닫힘
- "취소"는 운영진만 — 사용자 자가 취소 X (1차 운영 단순화)

### UX-6: 즉각 피드백 (Instant Feedback)

**원칙:** 사용자 액션 후 *200ms 안에 시각 피드백*. SSE 상태 변경은 *깜박이며 갱신*.
**근거:** 페르소나 ① (들떠있음, 빠른 응답 기대)
**구현:**
- 카트 담기 = 50-100ms scale 펄스
- "주문하기" 클릭 = 즉시 spinner + CTA 비활성화
- SSE 단계 변경 = 카드 0.5초 형광 옐로 깜박 → 새 카피
- 도장 찍힘 = 150ms

### UX-7: 외부인 1초 인지 (External 1-Sec Recognition)

**원칙:** 외부인이 *학번 입력 화면*에서 1초 안에 *"학번 없음" 체크박스*를 발견해야 함.
**근거:** 페르소나 ② (어색함, 학번 없음)
**구현:**
- 학번 input 위에 큰 체크박스 ("□ 학번 없음 (외부인)")
- 체크 시 학번 input 즉시 비활성화 + 회색 처리
- 체크박스 = 48px hitbox + 형광 옐로 외곽

### UX-8: 운영진 = 정보 밀도 + 키보드 (Admin Density + Keyboard)

**원칙:** 운영진 화면은 *정보 밀도 최대*. 마우스 클릭 줄이고 *키보드*로 완료 가능하게.
**근거:** 페르소나 ③ (본부, 30건 동시) + 페르소나 ④ (손 더러움)
**구현:**
- 본부 대시보드 = 5단계 카드 한 화면 동시
- 카드 내 텍스트 = 14px, 정보 위계 *학번·금액·시간*만 강조
- 마스코트 사용 X (정보 노이즈)

**MVP 키보드 단축키 — 4종 (2026-05-13 축소 결정 D):**

| 키 | 동작 | 비고 |
|---|---|---|
| `Enter` | 포커스된 카드의 *주 액션* 수행 | 대시보드: 이체 확인 / 주문 상세: 다음 단계 전이 |
| `Esc` | 모달·상세 닫기 / 검색 초기화 | 브라우저 기본과 일치 |
| `Tab` / `Shift+Tab` | 카드·필드 순회 | 브라우저 기본 |
| `?` | 단축키 도움말 모달 | 운영 가이드 |

**Phase 2 후보 (1차 운영 후 결정):**
- `1` ~ `9` 주문번호 점프 / `0` 직접 입력 모달
- `g d`·`g t`·`g s`·`g m` chord 네비게이션
- `h` 포커스 카드 빠른 HOLD 전이

**축소 근거 (2026-05-13 D 결정):**
- 5-10명이 2일만 쓰는 시스템에 vim chord 학습 비용 > 가치
- chord 디스패치 로직 X (Alpine.js + 표준 이벤트만)
- 본부 폭주 시 가속화는 FIFO 정렬 + Enter로 충분
- 1차 운영 후 *진짜 빈번한 동작*만 골라 Phase 2 추가

**구현 메모:**
- 입력 필드(textarea·input) 안에선 단축키 비활성화 (텍스트 입력 우선)
- `?` 모달은 4종만 안내 (Phase 2 후보도 함께 명시 — 운영진이 *향후 추가 요청* 인지)

### UX-9: 회복 가능한 실패 (Recoverable Failure)

**원칙:** 모든 실패는 *왜 실패했는가 + 어떻게 회복할 수 있는가* 명시. 막다른 골목 X.
**근거:** 페르소나 ① (불안), ARCHITECTURE.md §7
**구현:**
- 쿠폰 거부 = 사유 + 재입력 가능 (예외: 중복 사용은 *이미 사용함* 안내)
- 이체 정보 매칭 실패 (HOLD) = 사용자 정보 수정 후 재제출 가능
- 네트워크 끊김 (SSE) = "다시 연결 중..." + 자동 재연결
- 메뉴 품절 (주문 도중) = "이 메뉴는 품절되었어요. 다른 메뉴로 변경해주세요" + 카트로 돌아감
- 404·500 = 마스코트 + "부스에 직접 문의해 주세요"

### UX-10: 인수인계 가능한 시스템 (Transferable Across Years)

**원칙:** *내년 학생회*가 봐도 1차 운영 시 어떻게 사용했는지 알 수 있게. 운영진 화면은 *툴팁·인라인 가이드* 풍부하게.
**근거:** Dream State (선후배 인수인계 자산화)
**구현:**
- 본부 대시보드 첫 진입 시 *간단한 워크 스루* (선택, 1회)
- 정산 화면 = 각 항목 옆 (?) 버튼 → 의미·계산식 안내
- "오늘 정산 마감" 버튼 옆 = "ADR-012 가드: 진행 주문 0건일 때만" 짧은 안내

### 5.1 학생 사용자 — 정상 흐름

17:25         부스 줄 진입                       (없음 — 시스템 외부)                기대감 +2
17:26         QR 스캔 → 메뉴 페이지              메뉴 + 인기 랭킹 즉시 표시           탐색 시작
17:28         메뉴 2개 선택 → 카트                카트 카운트 펄스 (피드백)            결정 +1
17:29         "주문하기" 탭                      → 주문 폼 (학번·이름·테이블·수령)     집중

17:49         음식 받음                          수령 담당자 "전달 완료" 클릭          만족 +5

**개선 포인트:**
- 17:48 직전 (대기 10분 구간) — 마스코트가 *idle 애니메이션*으로 살아있음 신호 (멈춰 보이지 않게)
- 17:48 READY 알림 — 화면 *진동(Vibration API)* + 형광 옐로 깜박 (호명 못 들을 수 있음)

### 5.3 본부 담당자 — 이체 확인 흐름

───────────  ──────────────────────────────────  ────────────────────────────────  ──────
17:30        대시보드에 *주문 중* 카드 5개         (대기)                              관찰

## 6. 상태 원칙 (Empty / Loading / Error / Success)

### 6.1 빈 상태 (Empty States)

**원칙:** *이유 + 다음 액션*. 사용자가 "왜 비어있나, 뭘 해야 하나" 1초 안에 파악.

| 상황 | 기획 | 카피 |
|---|---|---|
| 카트 비어있음 | 마스코트 (기본) + 카피 + "메뉴 보기" CTA | "줍기 시작해 보세요! 인벤토리가 비어있어요." *(2026-05-13 PUBG 인벤토리 라벨 G11)* |
| 인기 랭킹 데이터 부족 | 카피 fallback | "🔥 학생회 추천 BEST" (ADR-017, 정적 단일 카피로 통일 2026-05-13 E) |
| 메뉴 검색 결과 없음 | (검색 미도입 — N/A) | — |
| 본부 대시보드 *주문 0건* | "오늘 첫 주문을 기다리는 중!" + 시간 | 친근 |
| 본부 대시보드 *CLOSED 상태* (G13) | 큰 형광 옐로 "🚀 장사 시작" CTA + 영업 시간 안내 | "16:30 오픈 예정 — 장사 시작 버튼을 눌러주세요" |
| 정산 화면 데이터 없음 | "오늘 정산 마감 후 표시됩니다" | 안내 |
| 쿠폰 사용 내역 0건 | "아직 사용된 쿠폰이 없어요" | 사실 |
| **사용자 영업 외 안내 (G13 신규)** | 마스코트 (기본) + 운영 일정 + "🔄 새로고침" | "🔒 영업 시간이 아니에요. 5/20·21 16:30 오픈" |
| **부스 미니맵 모달 약도 미수령 (G12)** | CSS 그리드 + 본인 테이블 형광 옐로 강조 | "테이블 #7 — 약도 준비 중" |

**금지:** 빈 화면, "No data" 영문, 빈 사각형.

### 6.2 로딩 상태 (Loading States)

**원칙:** *예상 대기 < 500ms*면 표시 X. *500ms-3초*면 spinner + 라벨. *3초+*면 진행 표시 + 취소 옵션.

| 상황 | UI |
|---|---|
| 페이지 진입 | spinner X (서버 렌더라 즉시) |
| API 호출 (~500ms) | CTA 버튼 비활성 + 안 글씨 회색화 |
| ZIP 다운로드 (수~수십 초) | 모달 + progress bar + "백업 만드는 중... 닫지 마세요" + 추정 X |
| SSE 첫 연결 | "조리 현황 연결 중..." 1줄 (재연결 시에만 표시) |
| 메뉴 이미지 로드 | placeholder = fallback 이모지·마스코트 즉시 표시, 이미지 도착 시 cross-fade |

**금지:** 스켈레톤 로더 (이 시스템 빠름·간단), 무한 spinner without context, 진행 표시 없는 30초+ 작업.

### 6.3 오류 상태 (Error States)

**원칙:** *사실 + 회복 경로 + 마스코트 (가벼운 에러만)*. 사용자 책임처럼 보이는 카피 금지.

| 카테고리 | 카피 패턴 | 예시 |
|---|---|---|
| 입력 오류 | "X은(는) Y해야 해요" + 재입력 강조 | "학번은 9자리 숫자입니다." |
| 권한 오류 | "이 정보는 다른 분 거예요" + 안내 | "주문자 정보가 일치하지 않아요." |
| 일시적 오류 (네트워크) | "잠시 후 다시 시도해 주세요" + 자동 재시도 | "연결이 끊어졌어요. 다시 연결 중..." |
| 비즈니스 룰 위반 | 사실만 + 다음 액션 | "이 학번은 이미 쿠폰을 사용했어요. 즐거운 축제 보내세요!" |
| **쿠폰 학과 검증 실패 (학과 코드 37 X)** | 사실 + 안내 | "이번 쿠폰은 컴퓨터모바일융합과 학생 전용입니다." *(2026-05-13 ADR-019 변경)* |
| **영업 외 (HTTP 423 BUSINESS_CLOSED)** | 영업 외 안내 화면으로 redirect (G13) | "🔒 영업 시간이 아닙니다. 5/20·21 16:30 오픈" |
| 시스템 오류 (500) | 마스코트 (취소 변형) + 부스 안내 | "잠시 시스템에 문제가 생겼어요. 부스에 직접 문의해 주세요." |
| 404 | 마스코트 + 홈으로 돌아가기 | "이 페이지는 임무에서 사라졌어요." |

**금지:**
- "Unknown error" 같은 영문·기술 메시지
- 사용자 책임 카피 ("잘못된 입력입니다")
- 마스코트가 *결제·금액 에러*에 등장 (UX-3)
- 토스트 알림 (UX-1: 모바일 부스 환경 인라인이 정답)

### 6.4 성공 상태 (Success States)

**원칙:** *명확한 완료 신호 + 다음 액션*.

| 상황 | UI |
|---|---|
| 주문 접수 | 도그태그 떨어짐 + "WINNER WINNER" + "조리 현황 보기" CTA |
| 이체 확인 (사용자) | 화면 상단 인라인 메시지 + 조리 현황판 자동 진입 |
| 이체 확인 (본부) | 카드가 다음 컬럼으로 슬라이드 + 짧은 형광 옐로 깜박 |
| 정산 마감 성공 | "5/20 정산 마감 완료" + ZIP 다운로드 강조 |
| ZIP 다운로드 완료 | "다운로드 완료. USB 보관 잊지 마세요!" 인라인 |

  경고     "[⚠️ 5분+ 지났어요 — 사용자가 기다립니다]" (16:30 이후 5분+ CLOSED 시)
```

> 위 카피는 기획서 §10.4 그대로. 운영 중 변경 금지 (인쇄물·QR 링크 동기화 부담).

### 7.5 카피 작성 일정·위치 (구현 전 작업)

**문제:** 톤 가이드와 마스코트 9종 카피만 있고, *13개 P0 화면별 실제 카피 70-130개*는 어디에도 없음. 구현 중 즉흥 작성 시 *톤 일관성 깨짐*·*시간 압박*.

**해결책:** 구현 *시작 전* 별도 문서 `docs/COPY_DRAFT.md`로 모아 작성.

**일정 (오늘 5/10 기준, D-day 5/20):**

```
5/10 ~ 5/12 (3일)   카피 1차 드래프트 → docs/COPY_DRAFT.md
                    ├─ 사용자 화면 6개 (메뉴·카트·주문폼·도그태그·이체·조리현황)
                    └─ 관리자 화면 7개 (로그인·대시보드·주문상세·이체확인·메뉴·정산·ZIP)
   ↓
5/13              카피 톤 일관성 자체 검증 (§7.2 체크리스트)
                  + 학생회 운영진 1명 읽고 "친근체 OK?" 검토
   ↓
5/14 ~          구현 시작 (사용자 신호 후)
                구현 시 COPY_DRAFT.md 그대로 활용 — 즉흥 X

**작성자:** 1차는 개발자 작성 (Claude Code 활용). 학생회 운영진이 *읽고 친근체 검토*.

**COPY_DRAFT.md 구조 (예약):**

> 이 일정은 D-day 압박을 고려한 *제안*. 사용자가 *구현 시작* 신호 시점에 우선 COPY_DRAFT.md 작성 → 그 후 코드.

## 8. 접근성 사용자 시나리오

### 8.1 시나리오 ① — 시각 장애 사용자 (스크린리더)

**상황:** 박서연의 친구가 *VoiceOver* 사용. 메뉴 → 주문 → 도그태그 → 조리 현황 흐름.

| 검증 포인트 | 통과 기준 |
|---|---|
| 메뉴 카드 음성 안내 | "후라이드 치킨, 18,000원, 추천 메뉴, 카트 담기 버튼" 한 단위로 |
| 카트 담은 후 피드백 | `aria-live="polite"`로 "카트에 담겼습니다" 음성 |
| 학번 input | label "학번 9자리 숫자" 음성 + 형식 오류 시 음성 안내 |
| 주문 완료 = 도그태그 | `<output role="status">` "주문번호 17번 발급되었습니다" 음성 |
| 조리 현황판 SSE | 단계 변경 시 `aria-live` 음성 ("조리 중입니다") |

### 8.2 시나리오 ② — 키보드만 사용 (마우스 X)

**상황:** 본부 담당자가 마우스 사용 못 하는 상황 (손 다침).

| 검증 포인트 | 통과 기준 |
|---|---|
| Tab으로 카드 순회 | 시각적 포커스 명확 (형광 옐로 외곽) |
| Enter = 이체 확인 | 카드 포커스 후 Enter로 액션 발동 |
| Esc = 카드 닫기 | 모달도 동일 |
| 1-9 = 주문번호 점프 (Phase 2) | 본부 대시보드 가속화 |

### 8.3 시나리오 ③ — 저시력 (확대 필요)

**상황:** 외부인 김민호가 노안. 시스템 폰트 크게 봐야 함.

| 검증 포인트 | 통과 기준 |
|---|---|
| 폰 OS 텍스트 크기 200% | 레이아웃 깨지지 않음, 가로 스크롤 X |
| 메뉴 카드 가독성 | 16px → 32px 변환되어도 카드 구조 유지 |
| 폼 라벨 | 입력 필드 위 명시적 (placeholder 라벨화 X) |

### 8.4 시나리오 ④ — 시끄러운 환경 (음성 안 들림)

**상황:** 부스 앞 줄에서 *호명 못 들음*.

| 검증 포인트 | 통과 기준 |
|---|---|
| READY 단계 시각 신호 | 화면 전체 형광 옐로 깜박 + 도그태그 강조 |
| 진동 알림 | Vibration API (Android) — *짧은 진동 2회* |
| 폰 화면 끄고 있을 때 | (브라우저 한계) — Service Worker 미도입이라 X. 사용자가 폰 화면 켜놓도록 안내 |

### 8.5 시나리오 ⑤ — 한 손만 자유 (다른 손 음식·짐)

**상황:** 페르소나 ① — 친구 손·메뉴판·핸드폰을 동시에.

| 검증 포인트 | 통과 기준 |
|---|---|
| 모든 CTA가 화면 하단 1/3에 | UX-1 |
| 외부 클릭으로 모달 닫기 | 정확한 X 버튼 누를 필요 X |
| 텍스트 입력 최소화 | 학번·이름·은행 외 자유 텍스트 X |

### 8.6 시나리오 ⑥ — 외부인이 친구·가족과 조리 현황 공유 (의도된 동작)

**상황:** 페르소나 ② 김민호(외부인 학부모)가 *주문 완료 화면 URL을 카톡으로 가족에게 공유*. 가족이 클릭하면 동일 조리 현황판 접근.

**의도된 동작 (정책):**
- ✅ **의도된 부수 가치:** 가족·친구가 *함께 진행 확인* 가능 (Apple Find My 같은 친근한 공유). 1차 운영 단순함 우선.
- 토큰만으로 인증 = URL 공유 시 누구든 조리 현황 조회 가능
- *정보 노출 위험 저:* 학번·이체정보·계좌번호는 조리 현황판에 노출 X (단계 카피만)
- *학생 사용자는 다른 시나리오:* 학번 매칭이 인증 키라, *학번 노출 X* 안 하면 친구 조회 불가 → 도그태그 SNS 공유 시 *학번 가리기* 기본 권장 (운영 가이드)

**비-의도 시나리오 (1차 운영 미도입):**
- ❌ 토큰 1회용 (구현 부담)
- ❌ IP 핀닝 (학교 와이파이 환경 부적합)
- ❌ 시간 제한 (24h는 이미 ADR-021)

**Phase 2 후보:**
- "주문 공유" 명시적 버튼 (의도 공유) vs 자동 차단 (URL 토큰 강화) — 1차 운영 후 결정.

## 9. UX 측정 지표 (1차 운영 후 회고용)

5/20-21 운영 후 *다음 해 학생회*가 활용할 정량·정성 지표.

| 지표 | 측정 방법 | 목표 |
|---|---|---|
| 주문 한 건 처리 시간 | 메뉴 진입 → 도그태그 발급까지 (analytics 미도입이면 사용자 인터뷰) | 30초 → 10초 |
| 주문 누락률 | 정산 시 *주문 시도 vs 완료* 비율 | 5% → 1% |
| "내 주문 어디?" 문의 | 부스 운영진 카운트 | 수십 건 → 0 |
| 정산 소요 시간 | 마감 → ZIP 다운로드 + 통장 대조까지 | 반나절 → 1시간 |
| 운영진 만족도 | 운영 후 5점 척도 | 4점+ |
| 사용자 만족도 | 운영 중 "음식 받음" 직후 짧은 1점 평가 (Phase 2) | 4점+ |
| 쿠폰 위조 시도 | rejected_coupons 테이블 | 검증·기록만 |
| 조리 현황판 사용률 | 주문 완료 → 조리 현황판 진입 비율 | 70%+ |

## 10. 차후 검토 (Phase 2 후보)

- 짧은 1점 평가 ("음식 받음" 직후) — 1차 운영에선 부담
- 다국어 (영어 첨가) — 외부 손님 대비
- Service Worker 푸시 알림 (READY 단계) — 브라우저 한계 검증 필요
- 사용자 자가 취소 (주문 후 1분 내) — 1차 운영은 운영진만 취소
- **키보드 단축키 chord (`g d`·`g t` 등) + 숫자 점프 + `h` HOLD** — 2026-05-13 D 결정으로 MVP에서 빠짐. 1차 운영 후 *진짜 빈번한 동작*만 골라 추가
- **부스 미니맵 PUBG 톤 줌·드래그** — G12 미니맵 모달은 MVP에서 정적 (모달 진입/닫기만). 줌·드래그 인터랙션은 1차 운영 후
- **테이블별 점유 상태 실시간 표시** (운영진만) — 미니맵 확장
- **영업 시작 시각 카운트다운** (영업 외 안내 화면) — "16:30 OPEN까지 17:32 남음" 형식
- **G14 일회성 명시:** *내년 학생회 인수인계 기능*은 Phase 2 후보에서 제외 (`project_one_time_service.md` 메모리)
