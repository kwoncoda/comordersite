## 2. 검토 기준별 답

### 2.1 이 문제가 진짜 중요한 문제인가? → **조건부 YES**

PRD.md §2.1이 열거한 8가지 현장 문제를 *임팩트 강도* 순으로 재분류:

| 문제 | 진짜 임팩트 | 시스템이 해결 |
|---|---|---|
| **정산 자료 자동 집계 (반나절 → 1시간)** | ⭐⭐⭐ HIGH — 학생회 회의 자산 | ✅ 진짜 해결 |
| **DB 장애 시 정산 근거 보존 (ZIP)** | ⭐⭐⭐ HIGH — 작년 망한 부스 학생회면 강하게 공감 | ✅ 진짜 해결 |
| **종이 영수증 분실·이중 처리** | ⭐⭐ MID — 실제 빈번 발생 | ✅ 해결 |
| **운영진 목 갈라짐 (호명)** | ⭐⭐ MID — 8시간 부스 운영 시 진짜 문제 | ⚠️ 부분 (시스템이 호명 자체를 대체 X, 진동만) |
| **메뉴 품절 지연 알림** | ⭐⭐ MID — 빈번 | ✅ 해결 |
| **"내 주문 언제?" 문의** | ⭐ LOW — 호명 잘 들리면 큰 문제 X | ⚠️ "조리 현황판" 가치는 *대기 길어질 때만* 의미 |
| **큰 소리 주문 시끄러움** | ⭐ LOW — 부스 환경 자체가 시끄러움 | ❌ 본질적 해결 X |
| **1학년 환영 쿠폰 운영** | ⭐ LOW — 1,000원, 대상자 ~120명. 실제 사용률 검증 X | ⚠️ 기능은 있으나 *왜 필요*한지 약함 |

**핵심 가치 (CEO 시각):**

1. **정산 자동화** — 가장 강한 가치. 학생회 회계 1명의 반나절 → 1시간으로. 이게 *유일하게* 측정 가능한 정량 이득.
2. **데이터 보존** — DB 손상 시 정산 근거 ZIP. 사고가 안 나면 가치 0이지만, *날 때*의 가치는 무한대.
3. **주문 흐름 자체** — 부분적 가치. *줄이 길 때만* 의미. 줄 없으면 시스템이 도리어 마찰.

**Inversion (Munger): "이 시스템 없이도 잘 굴러갈까?"**
→ 작은 부스라면 잘 굴러감. 줄이 30명 이상 동시 폭주하는 *피크 시간*에만 시스템의 가치가 *결정적*.

**결론:** 문제는 진짜다. 단 *피크 폭주 시간 + 정산 자동화*가 핵심 가치이고, 나머지 (조리 현황판·인기 랭킹·쿠폰)는 *보조*다. 의사결정 시 이 가치 위계를 기억할 것.

### 2.2 사용자가 실제로 쓸 이유가 있는가? → **YES (단, 동기 설계 필요)**

**비교 분석:**

| 시나리오 | 직접 주문 (소리) | 시스템 (모바일) |
|---|---|---|
| 줄 0명 | **5-10초** ⭐⭐⭐ | 60-120초 |
| 줄 5명 | 30-60초 | **60-120초** (병렬) |
| 줄 10명 이상 | 3-10분 | **60-120초** ⭐⭐⭐ |
| 외부인 (학번 X) | 자연스러움 | 체크박스 마찰 |
| 시끄러움·복잡 | 헷갈림 | 깔끔 |

**진짜 사용 동기 3종:**

1. **줄 우회** — 줄이 길면 시스템이 유일한 합리적 선택. *핵심 유스케이스*.
2. **인스타 공유 (도그태그 + WINNER WINNER)** — 학생들이 *재미로* 시스템 사용 → SNS 공유. 단순 주문 도구 X, *경험* 상품. 이게 *진짜 차별화*.
3. **친구와 함께 (외부인 공유 URL)** — UX_STRATEGY §8.6에 명시. 가족·친구가 함께 조리 현황 보기. 사회적 부산물.

**리스크: 시스템 우회 (Bypass)**

학생이 부스에 와서 "후라이드 하나요!"라고 외치면 *그게 더 빠름* (줄 없을 때).
→ **운영 가드 필요**: 부스 운영진이 *현장 주문을 거부*하고 "QR 스캔해 주세요"로 안내해야 함. 시스템 우회 허용 시 도리어 *주문 누락·이중 처리*가 *늘어남*.
→ **UX 가드**: 부스 입구 QR + "줄 서지 마세요, 폰으로 주문!" 안내판.
→ **PRD 미반영**: 이 운영 정책이 PRD에 명시되어 있지 않음. **추가 권장**.

**도그태그 SNS 가치 (정량 추정):**

- 5/20-21 양일 ~1,000-1,800 주문 가정
- 인스타 공유율 5% 가정 (보수적) → 50-90명 공유
- 친구 노출 10명/인 → **500-900 노출**
- 학내 인지도 효과: 작년 학생회 부스와 *차별화*

이 SNS 효과 하나만으로도 시스템 투자 정당화 가능 (학생회 마케팅 자산).

**결론:** 사용 동기는 있다. *줄 우회 + SNS 공유* 둘이 핵심. 단 **운영 정책 (현장 주문 거부)**를 PRD에 명시해야 시스템 우회 차단.

### 2.4 반드시 빼야 할 기능이 있는가? → **반드시 빼야 할 건 없음. 단, 축소 가능 후보 7종.**

> **메모: 사용자 메모리에 "범위 축소 제안 자제"가 있음. 이 섹션은 사용자가 *명시적으로 요청*했기에 작성. 후보 제시는 했으나 *cherry-pick 권한은 사용자*. 압박 X.**

**축소 후보 (효과 큰 순):**

#### 후보 A: 자동 ZIP 스냅샷 주기 30분 → 2시간

- **현재:** 30분 × 6 회전 = 3시간 보존창 (ADR-022)
- **변경:** 2시간 × 6 회전 = 12시간 보존창
- **이득:** CPU·디스크 부담 1/4. 2일 운영에 12시간 보존이면 충분.
- **위험:** 데이터 손실 최악값 30분 → 2시간 (KPI 변경 필요).
- **추천:** ✅ **수용 권장** — 2일 운영 환경에 30분 주기는 over-engineering.

#### 후보 B: 마스코트 5종 → 2종 + 이모지 fallback

- **현재:** 기본·출동·조리·도착·취소 (DESIGN.md §10.1)
- **변경:** 기본·조리 2종 + 도착·취소는 이모지 (🎉·❌)
- **이득:** SVG 디자인 작업 60% 절감 (~6-10시간).
- **위험:** Memorable thing 약화. 도그태그·WINNER WINNER가 핵심이라 보존됨.
- **추천:** ⚠️ **사용자 결정** — 마스코트 보유 자산 활용 욕구 있을 시 5종 유지.

#### 후보 C: 도장 5종 SVG → CSS 텍스트 stamp

- **현재:** SOLD OUT·RECOMMENDED·PAID·READY·DONE SVG (DESIGN.md §8.1)
- **변경:** CSS `transform: rotate(-3deg)` + 폰트 + border + Black Ops One
- **이득:** SVG 5종 작업 → 0. CSS class 1개 + 변형 5개.
- **위험:** 도장 텍스처 손맛 약간 감소. Risk 1 (ADR-026 §8) 정도는 보존.
- **추천:** ✅ **수용 권장** — 코드로 충분히 표현 가능. 사용자가 "진짜 도장" 톤 고수면 SVG.

#### 후보 D: 키보드 단축키 12종 → 4종

- **현재:** Enter·Esc·Tab·1-9·0·g+d/t/s/m·h·? (UX-8)
- **변경:** Enter·Esc·Tab·? 4종만
- **이득:** chord 디스패치 로직 X, 도움말 모달 단순.
- **위험:** 본부 5분 누적 폭주 시 매칭 속도 ↓ (1초/주문 × 30 = 30초 누적 ≈ 무시 수준).
- **추천:** ✅ **수용 권장** — 5-10명이 2일만 쓰는 시스템에 vim chord 학습 비용 > 가치. 1차 운영 후 회고에서 빈번 동작 골라 Phase 2.

#### 후보 E: 인기 랭킹 동적 카피 → 정적 "BEST"

- **현재:** "압도적 1위·추격 중·🚀 급상승" 규칙 기반 카피 (ADR-017)
- **변경:** "🔥 학생회 추천 BEST" 한 줄 고정 + TOP 3 메뉴명·판매수만 표시
- **이득:** 카피 생성 로직 100여줄 절감. 운영진 ON/OFF 토글도 불필요.
- **위험:** Social proof 효과 약화 (실제로 그 효과가 얼마나 큰지 검증 X).
- **추천:** ⚠️ **사용자 결정** — "social proof로 메뉴 선택 가속화"가 가설인데 1차 운영에서 검증 X. 데이터 없이 *직감*으로 추가한 기능이라 빼도 무방.

#### 후보 F: 카모 패턴 SVG → CSS gradient

- **현재:** 10×10 픽셀 카모 SVG (DESIGN.md §8.2)
- **변경:** CSS `background: linear-gradient(...)` + olive/sand/earth 색상 mix
- **이득:** SVG 작업 0. 토큰 변경 1줄로 색상 조정.
- **위험:** 디지털 카모 톤 약간 약화 (5-10% opacity 텍스처용이라 잘 안 보임).
- **추천:** ✅ **수용 권장**.

#### 후보 G: 부스 약도 자체 제작 → "부스 입구에 큰 종이 약도"

- **현재:** `/map` P1 화면 (사용자 약도 이미지 수령 후)
- **변경:** 시스템에서 약도 화면 *제외*. 부스 입구에 종이 안내판.
- **이득:** 약도 이미지 수령 의존성 X, P1 화면 1개 제거.
- **위험:** 사용자가 QR 스캔 후 *부스 위치* 모를 수 있음.
- **추천:** ⚠️ **사용자 결정** — 학생회 부스는 어차피 학내 잘 보이는 곳에 있을 가능성 ↑.

**결론:** *반드시* 빼야 할 건 없다. 단, A·C·D·F는 *얻는 게 많고 잃는 게 작다* (low-risk 축소). B·E·G는 사용자 결정.

### 2.5 첫 출시에서 가장 중요한 기능 1개 → **본부 매칭 흐름 (이체 확인)**

**후보 비교:**

| 후보 | 가치 | 우선순위 |
|---|---|---|
| 메뉴 + 카트 + 주문 폼 | 입력 | 진입점, 작동 안 하면 시스템 미가동 |
| 도그태그 + WINNER WINNER | 절정 UX | 차별화의 핵심 |
| **본부 매칭 (이체 확인)** | **흐름 차단점** | **여기서 막히면 *나머지 모두 무용지물*** |
| 조리 현황판 + SSE | 안심 | 부수적, 호명으로도 OK |
| ZIP 백업 + 자동 스냅샷 | 정산 안정성 | D-day 후 가치 |

**왜 "본부 매칭"이 단일 가장 중요한 기능인가:**

```
사용자 흐름 단절 분석:

[메뉴] → [카트] → [주문폼] → [도그태그] → [이체] → [확인요청]
                                                        │
                                                        ▼
                                              ★ 본부 매칭 (병목)
                                                        │
                                                        ▼
[조리 시작] → [조리 완료] → [수령] → [DONE]
```

**본부 매칭이 막히면:**
1. 사용자: "내 주문 진행 안 됨" 불안
2. 조리: 매칭 전이라 시작 못 함
3. 결과: 사용자가 부스로 와서 직접 문의 → 시스템 가치 0
4. 매칭 backlog 5분 이상 = 부스 운영 마비

**다른 기능이 막혀도 시스템은 굴러간다:**
- 도그태그 모션 작동 X → 그냥 정적 텍스트로도 주문번호 인식 OK
- 조리 현황판 SSE 끊김 → 사용자는 부스에서 호명 듣고 받아감
- 인기 랭킹 X → 메뉴 직접 스크롤
- ZIP 백업 X → 정산 화면 수동 PDF 캡처

**단, 본부 매칭이 막히면 시스템 전체가 죽는다.**

**핵심 보강 권장:**

본부 매칭은 *기능*이 아니라 *워크플로*다. 다음 3개가 한 묶음:
1. `A-2 본부 대시보드` (5단계 카드 + 5초 폴링)
2. `A-3 주문 상세` 또는 `A-4 이체 확인 화면` (인라인 액션)
3. **운영 가드**: "5분 경과 노란 / 10분 빨간 + 진동" (F-A-008)

**추가 권장 (PRD 미반영):**
- **본부 백업 담당자 (Buddy System)** — 1차 본부 담당자 화장실·식사 시 무휴 처리 보장
- **카뱅 폰 분실 대비**: 미리 백업 폰 또는 PC 카뱅 웹 로그인 사전 설정
- **HOLD 처리 표준화**: HOLD 상태가 *방치*되지 않도록 3분 경과 시 자동 강조

이 3개가 *PRD §11.1 운영 리스크*에 명시되어야 함.

### 2.6 차별화 포인트가 충분한가? → **학생 시장에는 충분. 코드 차별화 X, 정서 차별화 O.**

**비교 분석 (학교 축제 부스 운영 시스템 대안):**

| 대안 | 비용 | 셋업 | 사용성 | 차별화 |
|---|---|---|---|---|
| 카카오톡 채널 + Google Form | 무료 | 5분 | 중 | 0 |
| 종이 영수증·번호표 | 종이값 | 0 | 낮 | 0 |
| 학교 본관 부스 시스템 (작년) | (모름) | (모름) | (모름) | (모름) |
| 외부 PG 사용 (포스 단말) | 비용↑ + 계약 | 1주 | 낮 | 0 |
| **이 시스템** | 개발 시간 + Docker 호스트 | 1주 | 중 | **PUBG theming + 도그태그 + SSE** |

**진짜 차별화 4종:**

1. **도그태그·WINNER WINNER theming** ⭐⭐⭐
   - 인스타 공유 동기 (학생 SNS 문화 적합)
   - 단일 차별화 포인트로 충분히 강함
   - 다른 부스에 *절대* 없음
2. **PUBG 인벤토리 "줍기·인벤토리" UI** ⭐⭐
   - 게임 친숙한 학생층 매력
   - 컴모융 학과 정체성과 맞음 (게임 개발 적성)
3. **조리 현황판 SSE** ⭐⭐
   - 다른 부스에 없음
   - 사용자 안심 효과
4. **자동 정산 + ZIP** ⭐⭐
   - 학생회 회의 자산 자동화
   - 인수인계 가치 (Dream State)

**차별화의 "왜" (Munger Inversion):**

> "다른 사람이 이걸 5분 만에 따라 만들 수 있나?"

- 카카오 채널 + 폼: ✅ 5분
- 우리 시스템: ❌ 최소 1주 (Docker + Express + SSE + 디자인)

→ 진입장벽 = 차별화. 단 1회성 학교 축제라 "지속적 차별화"는 의미 없음. **1회 적용 = 충분.**

**약점 (정직하게):**

- 코드 차별화 X — Express + EJS + Alpine.js는 평범. 누구나 쓰는 스택.
- 기능 차별화 약함 — *기능* 자체로는 일반 주문 시스템 + 약간의 게임 톤.
- 진짜 차별화는 *정서·SNS·문화 적합성*에 있음.

**결론:** 학생 시장에서 충분. 학교 외부 시장에서는 *해당 없음* (1회성 학교 운영이라 비교 대상 X). 차별화 보강 작업 필요 X.

### 3.2 단일 실패 지점 (Single Point of Failure)

**본부 매칭이 SPOF다.**

```
사용자 ─────▶ 시스템 ─────▶ [본부 매칭] ─────▶ 조리 ─────▶ 수령
                                  │
                                  └── 막히면 ─▶ 시스템 전체 마비
```

**완화 (PRD 미반영, 권장 추가):**

1. **Buddy System** — 본부 담당자 2명 교대 (식사·화장실 시 무휴)
2. **백업 카뱅 단말** — PC 카뱅 웹 미리 로그인 + 폰 백업
3. **HOLD 자동 강조** — 3분 경과 시 빨간 강조 (현재는 5분/10분만)
4. **수동 전환 가이드** — 시스템 다운 시 종이 영수증 fallback 절차

### 3.3 무시해도 되는 위험 (이미 차단됨)

- 가격 위변조 → ADR-020 Pattern B + 4 회귀 케이스
- SSE 30 동시 OOM → ADR-025 회귀 + `restart: always`
- 컨테이너 죽음 → Docker `restart: always` (~5초)
- 호명 못 들음 → 형광 옐로 + 진동 API
- 외부인 어색함 → UX-7 큰 체크박스 + 형광 옐로 외곽

## 4. PRD에 반영할 수정 사항 (요약)

상세 변경은 다음 PRD.md 커밋에서 확인. 여기는 *왜* 변경하는지만.

### 4.1 §4 핵심 가치 제안 — 북극성 시나리오 추가

**문제:** PRD §4.1에 5개 가치, §4.2에 6개 가치 나열됨. *우선순위·중요도 차이*가 명시되지 않음.

**변경:** 새 §4.6 "북극성 시나리오 (North Star)" 추가.
- 만약 시스템에서 *1개 흐름만 작동*해야 한다면 = 사용자가 메뉴 선택 → 본부 매칭 → 조리 → 수령
- 그 외 (도그태그 모션·인기 랭킹·조리 현황판 SSE·외부인 공유)는 *보조*

### 4.2 §11.1 운영 리스크 — 본부 매칭 병목 격상

**문제:** 현재 §11.1의 운영 리스크 5개 중 "본부 담당자 1명 병목"이 평범한 항목으로 나열. 실제로는 *단일 실패 지점*.

**변경:**
- §11.1 가장 위에 "🔴 단일 실패 지점: 본부 매칭" 추가
- 완화책 4종 (Buddy System / 백업 단말 / HOLD 자동 / 수동 fallback) 명시

### 4.3 §8 비범위 — "현장 주문 거부" 운영 정책 추가

**문제:** 사용자가 시스템 우회 (부스로 직접 와서 "후라이드 하나요!") 시 시스템 가치 ↓ + 주문 누락 ↑. PRD에 운영 정책 미명시.

**변경:** §8.3 운영 기능 또는 §11에 추가: "부스 운영진은 현장 주문을 거부하고 QR 스캔 안내. 시스템 우회 0%가 운영 정책."

### 4.4 §10.1 사용자가 제공할 자료 — fallback 우선 정책 명시

**문제:** "임시 fallback OK"라고만 적혀있고, 자산 미수령 시 *기본 동작 정의*가 약함.

**변경:** §10.1 표 아래에 fallback 정책 명시:
- 마스코트 미수령 → 헬멧 이모지 🪖 + 컬러 배경
- 도장 미수령 → 텍스트 stamp (CSS rotate + Black Ops One)
- 카모 패턴 미수령 → CSS gradient (올리브 + 사막 + 흙)
- 메뉴 일러스트 미수령 → 분류 이모지 (🍗 🍟 🥤)
- 약도 미수령 → `/map` 화면 P2로 격하 + 부스 입구 종이 안내

### 4.5 §7 KPI — "주문 처리 시간 10초"는 비현실적, 조정 권장

**문제:** §7 첫 줄 "30초 → 10초" — 시스템 흐름이 *최소 30-60초* (학번 + 폼 + 이체 + 확인 요청). 10초는 측정 불가.

**변경:** "30초 → 90초 (단, 부스 줄 0건. 줄 10명 시 폭주 처리 시간 10명 × 30초 = 5분 절감)"으로 재정의.
또는 KPI 자체를 "*처리량(orders/min)*"으로 바꾸기.
