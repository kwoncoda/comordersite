## 1. 비전 & 미션

### 1.1 비전 (Vision)

> **"축제 부스에서 주문 외치지 말고, 폰으로 주문하고 번호로 받아가세요."**

학교 축제라는 시끄러운 현장에서 사용자·운영진 모두 *조용히, 정확하게, 빠르게* 거래가 끝나는 경험을 만든다.

### 1.2 미션 (Mission)

- 종이·구두 주문에서 발생하는 **누락·오기·말다툼 0건**
- 운영진이 **주문 현황을 한 화면**으로 파악
- 축제 종료 후 **정산을 1시간 내** 완결 (이전: 반나절)
- 결제 시스템(PG) 도입 없이 **계좌이체 기반**으로 안전하게
- 사용자가 **"내 주문 어디?"** 불안 없이 본인 폰에서 확인
- 데이터 손실 시에도 **정산 근거 100% 유지**

### 1.3 서비스 정체성

| 항목 | 값 |
|---|---|
| 서비스명 | **오늘 저녁은 치킨이닭!** |
| 보조 카피 | **WINNER WINNER CHICKEN DINNER!** (PUBG 승리 메시지 차용 — 카지노 슬랭 일반어, ADR-013 가드레일) |
| 별칭 | 치킨이닭 / 오저치 |
| 운영 주체 | 컴퓨터모바일융합과 학생회 |
| 마스코트 | 군용 헬멧·아머·소총을 든 흰 곰 (`docs/웹 로고.png`) |
| 시각 메타포 | **PUBG 인벤토리 / 파밍** (변경 2026-05-13) |
| Memorable Thing | 주문번호 받는 순간 = *도그태그 받는 느낌* (ADR-026 §1) |

> **변경 이력:** ADR-013에서 *"배민 UX + PUBG 비주얼 + 자체 마스코트"* 컨셉을 채택했고, ADR-026에서 시각 토큰을 구체화했다. 2026-05-13 사용자 결정으로 *비주얼의 PUBG 강도*를 키운다.

| 도그태그·WINNER WINNER·마스코트 | 보존 | **보존** |

**보존 원칙 (절대 깨면 안 됨):**
- ADR-020 Pattern B (서버 자체 계산)
- ADR-021 학번+이름 필수 + 외부인 체크박스
- ADR-012 정산 마감 가드 (진행 주문 0건일 때만)
- ADR-023 Docker compose 배포
- 마스코트 군용 헬멧 곰 사용 위치 (조리 현황판·빈 상태·에러 화면만)
- 도그태그 주문번호 메모러블 모션
- 형광 옐로 CTA (부스 호명 신호)

### 1.5 한 줄 소개

학교 축제 부스에서 모바일 웹으로 주문 → 계좌이체 → 운영진 통장 매칭 → 조리 → 도그태그 호명 수령까지를 *한 줄에* 끝내는 시스템.

## 2. 비즈니스 컨텍스트

### 2.1 현장 문제

| 현장 문제 | 본 시스템의 해결 방식 |
|---|---|
| 큰 소리 주문이 시끄럽고 헷갈림 | 폰에서 메뉴 선택 → 주문번호 발급 |
| 종이 주문서 분실·이중 처리 | 모든 주문을 DB 1곳에 모음 (`orders` 테이블) |
| 메뉴 품절을 늦게 알림 | 운영진 토글 → 사용자 화면 즉시 반영 |
| 정산할 때 종이 영수증 더미 | 자동 집계 화면 + 관리자 수동 트리거 |
| 컴모융 학생 환영 쿠폰 운영 어려움 | 학번 패턴 검증 (학과 코드 `37`) + 중복 차단 |
| 입금 매칭이 어려움 | 사용자가 이름·은행 신고 → 운영진 통장 대조 |
| 사용자가 "조리 완료됐나?" 모름 | 조리 현황판 — 본인 진행 상황 직접 확인 |
| DB 장애·서버 종료 시 정산 근거 소실 | 정산 ZIP 백업 + 30분 자동 스냅샷 |

### 2.2 왜 모바일 웹인가

- 학교 축제 사용자 100% 모바일 폰 (ADR-008)
- 앱 설치 마찰 0 — QR 스캔으로 즉시 진입
- 외부 PG 도입 불필요 (ADR-009)
- 운영진 인수인계 자산화 (Docker compose + 단일 서버, ADR-023)

### 2.3 왜 PG 없이 운영하는가

학교 축제 규모·계약 책임·정산 책임에서 PG 도입이 과함 (ADR-003, ADR-009). 사용자 자가 이체 + 사용자 신고 + 운영진 확인 3단계로 *노쇼 위험 0*에 가깝게 처리.

## 4. 핵심 가치 제안 (Value Propositions)

### 4.1 사용자 가치

1. **"주문이 1분에 끝난다"** — 메뉴 선택 → 학번/이름 → 도그태그 발급까지 30초 → 10초
2. **"내 차례 어디인지 폰에서 본다"** — 조리 현황판 SSE 실시간 단계 표시 (시간 약속 X)
3. **"외부인도 환영"** — 학번 없음 체크박스 1초 인지 → 외부인 진입 0초
4. **"WINNER WINNER 받는 의식"** — 도그태그 떨어지는 모션 + 형광 옐로 강조 = 단순 주문번호 X, 분대원 임무 부여받는 분위기
5. **"파밍하는 재미"** — PUBG 인벤토리 패러다임 — 메뉴 카드가 "땅에 떨어진 아이템", 카트가 "내 인벤토리"

### 4.2 운영진 가치

1. **"5단계 카드 한 눈에"** — 본부 대시보드에서 모든 주문 현황·경고 동시 모니터링
2. **"이체 확인이 빠르다"** — 통장 대조 4요소(이름·은행·금액·시각) + 키보드 Enter
3. **"마감 실수 안 생긴다"** — 진행 주문 0건 가드 (ADR-012)
4. **"정산 자료가 ZIP 1개"** — 매출·메뉴별 판매·쿠폰·이미지 모두 압축 (ADR-016)
5. **"DB 날아가도 안전"** — 30분 자동 스냅샷 (ADR-022)
6. **"내년 학생회가 봐도 안다"** — Docker compose + ADR 문서 + 단축키 카드 = 인수인계 자산

### 4.3 학생회 운영 가치

1. **목 안 갈라짐** — 호명을 *형광 옐로 + 진동 + 도그태그*로 대체
2. **노쇼 0** — 이체 확인 전 조리 시작 안 함 (ADR-003)
3. **운영진 5명으로 충분** — 본부 1·조리 2-3·수령 1 + 정산은 마감 후
4. **데이터로 다음 해 결정** — KPI 측정 가능

### 4.4 북극성 시나리오 (North Star) — *2026-05-13 PO/CEO 검토 반영*

> **만약 시스템에서 *1개 흐름만 작동*해야 한다면 = "사용자가 메뉴 선택 → 본부 매칭 → 조리 → 수령"이다.**

PO/CEO 검토(`docs/PO_REVIEW.md` §2.5)에서 식별한 *단일 절대 흐름*. 다른 기능은 모두 *보조*다:

| 절대 흐름 (1개) | 보조 (작동 안 해도 시스템 굴러감) |
|---|---|
| 메뉴 진입 → 카트 → 주문 폼 → 도그태그 발급 → 이체 안내 → **본부 매칭** → 조리 → 수령 (DONE) | 도그태그 떨어지는 *모션*, WINNER WINNER 카피, 인기 메뉴 동적 카피, 마스코트 5종 변형, 카모 패턴, 키보드 단축키 12종, 외부인 공유 URL |
| 만약 막히면 → 시스템 전체 마비 | 만약 빠지면 → 분위기·재미 손실. *작동은 함* |

**구현·운영 우선순위 함의:**

- **디자인 자산 미수령 시** → 보조 영역 fallback이 *기본*이고 절대 흐름 보장이 우선 (§10.1 fallback 정책)
- **일정 압박 시** → 절대 흐름 P0 외 모두 P1 격하 고려 (`docs/MVP_SCOPE.md` 축소 후보 검토)
- **운영 리허설 시 (D-1 5/19)** → 절대 흐름 end-to-end 1번 = 다른 100가지 검증보다 우선
- **D-day 사고 시** → 절대 흐름 복구가 최우선 (조리 현황판·인기 랭킹 등은 후순위)

이 북극성은 §11.1 단일 실패 지점 분석(본부 매칭 병목)과 직접 연결된다. 핵심 가치(§4.1·§4.2·§4.3)는 *수확물*이지만 *북극성은 절대 흐름의 가동 자체*다.

## 5. 비기능 요구사항 (NFR)

### 5.1 성능 (Performance)

| 지표 | 목표 | 출처 |
|---|---|---|
| 동시 처리 주문 (피크) | 10-30건 | G3 |
| 첫 페이지 로드 (3G 가정) | ≤ 2초 | UX-6 |
| API 응답 (주문 생성) | ≤ 500ms | UX-6 |
| SSE 연결 유지 | 30개 동시 무리 없음 | ADR-015 |
| ZIP 생성 (정산 + 자동) | ≤ 60초 | ADR-016/022 |
| 자동 ZIP 스냅샷 주기 | 30분 | ADR-022 |

### 5.2 보안 (Security)

| 영역 | 요구사항 | 출처 |
|---|---|---|
| 가격 무결성 | 서버가 menu_id·qty·coupon만 받아 자체 계산 (Pattern B) | ADR-020 |
| 학번 검증 | 9자리 숫자 + 정규식 `^\d{2}\d{2}37\d{3}$` (학과 코드 `37` 매칭) | 2026-05-13 변경 |
| 쿠폰 중복 차단 | `used_coupons` 학번 unique key | ADR-019/2026-05-13 |
| 관리자 인증 | PIN 로그인 + express-session | ADR-024 |
| PII 보관 | 정산 후 N일(7~14일) 자동 삭제 | ADR-021 |
| HTTPS | 학교 와이파이 환경 권장, 미적용 시 운영 가이드 명시 | SCREEN_STRUCTURE §7 |
| ZIP 파일 보안 | 관리자 PIN 권한 + 옵션 패스워드 보호 | ADR-016 |
| 보안 헤더 | helmet 8.x 기본 적용 | ADR-024 |
| OWASP Top 10 | A04 (Insecure Design) 기본 방어 — Pattern B로 차단 | ADR-020 |

**디자인 자산 fallback 우선 정책 (2026-05-13 PO/CEO 검토 반영):**

자산 미수령 시 *기본 동작*을 명시한다. fallback은 *예외*가 아닌 *기본 가정*이며, 절대 흐름(§4.4) 보장이 자산 디테일보다 우선한다:

| 미수령 자산 | Fallback 기본 동작 |

| 마스코트 5종 SVG | 헬멧 이모지 🪖 + 컬러 배경 (5단계 동일 이미지로 시작, 차후 추가 변형 교체) |
| 도장 5종 SVG | CSS `transform: rotate(-3deg)` + Black Ops One 폰트 + border (SOLD OUT·RECOMMENDED·PAID·READY·DONE 텍스트 stamp) |

| 메뉴 일러스트 8종 | 분류 이모지 (`치킨 🍗` + `사이드 🍟` + `음료 🥤`) + 컬러 배경 (ADR-006) |
| 부스 약도 이미지 | `/map` 화면 P2 격하 + 부스 입구 *종이 안내판* 운영 가이드 |

| 클라이언트 시각 조작 | 부정 정산 | 서버 시각만 사용 |

### 11.3 사용자 리스크

| 리스크 | 영향 | 완화 |
|---|---|---|
| 외부인이 "학번 없음" 못 찾음 | 이탈 | UX-7: 큰 체크박스 + 형광 옐로 외곽 |
| 사용자가 "확인 요청" 누락 | 조리 시작 안 함 | 빨간 강조 + 경고 카피 + 자동 redirect |
| 신고 이름 ≠ 통장 이름 | HOLD 누적 | UX-9 사용자 정보 수정 후 재제출 |
| 호명 못 들음 (시끄러움) | 음식 식음 | READY 화면 형광 옐로 깜박 + 진동 API |
| 사용자가 URL 공유 → 가족 조회 | 의도된 동작 | 정보 노출 위험 저 (단계 카피만 노출) — UX_STRATEGY §8.6 |

### 11.4 디자인 리스크 (2026-05-13 변경 관련)

| 리스크 | 영향 | 완화 |
|---|---|---|
| 군복 톤 ↔ 부스 조명 가독성 | 사용자 인식 저하 | *베이스 군복 + 메뉴 카드 밝은 흙색* 하이브리드. 본문 대비 4.5:1 유지 검증 |
| PUBG 인벤토리 패러다임 학습 비용 | 사용자 혼란 | 라벨에 "줍기/인벤토리" 사용하되 *행동은 배민 표준 동일* (탭 → 카트 → 결제) |
| "콜라 → 에너지 드링크" 시각만 매칭 시 인식 혼란 | 잘못된 주문 | 카드 상단에 *본명 명확히* 표기, 일러스트는 보조 |
| 디자인 변경 시간 부족 | D-day 지연 | 토큰 5개 값 조정 + 카드 변형 1개 추가만 — 추정 4-6시간 |
