### 2.3 핵심 결정 지점 (Decision Points)

| 시점 | 결정 | 분기 |
|---|---|---|
| `/checkout` | 학번 있나? | YES → 학번 입력 / NO → "학번 없음" 체크박스 (페르소나 ②로 분기) |
| `/checkout` | 쿠폰 사용? | 학과 코드 `37` 매칭 통과 → 1,000원 할인 / 실패 → 안내 + 입력 |
| `/checkout` | 수령? | 매장 → 테이블 번호 / 포장 → skip |
| `/orders/:id/complete` | 이체했나? | YES → "확인 요청" / NO → "조리 현황 보기" (대기) |
| `/orders/:id/transfer-report` | 다른 이름 이체? | YES → 추가 이름 입력 / NO → 주문자 이름 자동 |
| `/orders/:id/status` (READY) | 호명 들었나? | YES → 도그태그 보여줌 / NO → 진동·깜박이 흡수 |

## 3. 페르소나 ② — 외부인 분기

### 3.1 화면 흐름

```
/menu → /cart → /checkout
                  │
                  │ "□ 학번 없음 (외부인)" 체크박스 활성화
                  │ (UX-7: 큰 hitbox + 형광 옐로 외곽)
                  ▼
               학번 input 즉시 비활성화 + 회색
               이름만 입력 가능
                  │
                  ▼
               POST /api/orders { is_external: true }
                  │
                  │ 서버: 토큰 생성 (24h 만료)
                  │ Set-Cookie: token=xyz
                  ▼
               /orders/:id/complete?token=xyz
                  │
                  ▼
               /orders/:id/transfer-report (이름은 주문 시점 입력 사용)
                  │
                  ▼
               /orders/:id/status?token=xyz
                  │
                  │ 토큰 + 주문번호 매칭으로 인증
                  ▼
               (이후 정상 흐름과 동일 — SSE)
```

### 3.2 외부인 핵심 특징

- 학번 검증 *스킵*
- 토큰 = `Set-Cookie` + URL query 둘 다 (쿠키 잃었을 때 fallback)
- 만료 24시간
- 동명이인 충돌 위험 → 토큰만이 유일한 인증 키
- 가족·친구와 URL 공유 시 함께 조회 가능 (의도된 동작, UX_STRATEGY §8.6)
- 단계 카피만 노출 — 학번·이체 정보 노출 X

### 3.3 외부인 분기의 위험·완화

| 위험 | 완화 |
|---|---|
| 외부인이 "학번 없음" 못 찾음 | UX-7: 큰 체크박스 + 형광 옐로 외곽 |
| 토큰 쿠키 잃음 (브라우저 닫음) | URL query `?token=xyz` fallback |
| 가족 URL 공유 → 의도되지 않은 노출 | 단계 카피만 (개인정보 노출 0) |
| 동명이인 외부인 매칭 충돌 | 본부 매칭 시 *시각 + 금액* 4요소 검증 |

### 4.2 본부 담당자 핵심 동작 — 통장 매칭 4요소

| 요소 | 시스템 표시 | 통장 표시 | 매칭 기준 |
|---|---|---|---|
| 이름 | 주문자 이름 (또는 "다른 이름") | 이체자 이름 | *정확 일치* |
| 은행 | 사용자 선택 콤보 | 카뱅 알림에 명시 | 일치 |
| 금액 | 서버 계산값 | 카뱅 입금액 | *정확 일치* |
| 시각 | 신고 시각 | 카뱅 입금 시각 | *±5분* |

4가지 모두 일치 → "이체 확인" 클릭 → status=PAID + SSE push 사용자 화면.
1개라도 불일치 → "보류 + 사유" → status=HOLD → 사용자에게 정보 수정 요청.

### 5.2 조리·수령 담당자 특성

- 손에 기름·양념 → 큰 버튼·큰 글씨 필수
- 화면 자주 못 봄 → 진동 알림 X (사용자 측 진동만), 시각 강조
- 마우스 대신 *손가락 끝 터치*
- 한 명이 본부 + 조리 + 수령 겸할 수 있음 (운영진 부족 시)

### 7.1 상태 정의 (ADR-006, §6.1)

| # | 상태 | 한글 | 진입 조건 | 다음 가능 |
|---|---|---|---|---|
| 1 | `ORDERED` | 주문 중 | 사용자 주문 확정 (이체 전) | TRANSFER_REPORTED, CANCELED |

| 3 | `HOLD` | 보류 | 매칭 실패 (운영진) | TRANSFER_REPORTED (재시도), CANCELED |

| 5 | `COOKING` | 조리 중 | 조리 담당자 "조리 시작" | READY, CANCELED |

### 7.4 불법 전이 (서버 거부)

- ORDERED → PAID (이체 확인 요청 단계 스킵 불가)
- ORDERED → COOKING (입금 확인 전 조리 X)
- TRANSFER_REPORTED → COOKING (이체 확인 거치지 않음)
- DONE → 모든 상태 (음식 나간 후 변경 X)
- CANCELED → 모든 상태 (취소된 주문 되살리지 않음)
